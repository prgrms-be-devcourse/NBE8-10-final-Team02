## 1. 프로젝트 개요

### 1.1 프로젝트명
- AI 기술 면접 연습 플랫폼

### 1.2 한 줄 소개
- GitHub 활동과 문서형 포트폴리오를 한 번 등록하면, 자소서 작성부터 예상 질문 생성, 텍스트 기반 모의 면접, 결과 피드백 관리까지 한 흐름으로 이어서 사용하는 AI 기반 취업 준비 서비스

### 1.3 프로젝트 배경
- 기존 취업 준비는 GitHub 활동, 이력서, 수상기록, 지원 회사 정보를 여러 도구에 반복 입력해야 한다.
- 자소서 작성과 면접 준비가 분리되어 있어 데이터가 재사용되지 않는다.
- 개발자에게 중요한 repository 활동, commit 근거, 기술 선택 이유를 일반 채팅형 AI가 안정적으로 재사용하기 어렵다.

## 2. 문제 정의

### 2.1 기존 방식의 한계
- 포트폴리오 정보가 구조화되어 저장되지 않는다.
- 자소서와 면접 준비 흐름이 끊겨 있다.
- 결과 기록이 누적되지 않아 반복 학습이 어렵다.
- GitHub와 문서 포트폴리오를 함께 다루는 개발자 특화 준비 서비스가 부족하다.

### 2.2 해결하고자 하는 핵심 문제
- 포트폴리오를 재사용 가능한 데이터 자산으로 저장하고,
- 자소서 생성, 면접 질문 생성, 모의 면접, 결과 히스토리를 하나의 데이터 흐름으로 연결한다.

## 3. 프로젝트 목표

### 3.1 기능적 목표
- OAuth2 로그인
- GitHub repository 조회 및 사용자 본인 commit 수집
- 문서 업로드 및 텍스트 추출
- 지원 단위(application) 생성 및 source 선택
- 자소서 문항 관리와 초안 생성
- 면접 질문 세트 생성
- 텍스트 기반 모의 면접 세션 진행
- 결과 리포트와 히스토리 조회

### 3.2 서비스적 목표
- 첫 포트폴리오 등록 완료율을 높인다.
- 자소서 생성에서 면접 준비까지 자연스러운 전환 흐름을 제공한다.
- 실패를 숨기지 않고 부분 실패와 재시도를 사용자에게 명확히 안내한다.

## 4. 타겟 사용자

- 부트캠프 수료 후 취업 준비 중인 개발자
- 신입 개발자 취업 준비생
- 이직을 준비 중인 주니어 개발자
- GitHub와 프로젝트 경험을 근거로 맞춤형 준비를 하고 싶은 사용자

## 5. 서비스 핵심 가치

### 5.1 포트폴리오 재사용 구조
- repository, commit, 문서 추출 결과를 한 번 저장하면 이후 기능에서 반복 활용한다.

### 5.2 자소서와 면접 준비의 연결
- application이 회사명, 직무, 문항, source, 면접 질문 생성 기준을 하나로 묶는다.

### 5.3 개발자 특화 데이터 활용
- 일반 이력 정보뿐 아니라 GitHub 활동과 문서형 포트폴리오를 함께 사용한다.

### 5.4 결과 기록과 반복 학습
- 세션 결과를 점수, 태그, 코멘트 형태로 남겨 다음 연습에 반영한다.

## 6. 주요 기능

### 6.1 회원가입 및 인증
- GitHub, Google, Kakao OAuth2 로그인 지원
- 이메일/비밀번호 자체 회원가입은 MVP 제외
- 같은 이메일의 다른 소셜 로그인은 자동 병합하지 않음

### 6.2 포트폴리오 수집 및 저장
- GitHub URL 입력 시 public repository 조회
- GitHub OAuth 연동 시 private repository 접근 허용 가능
- 사용자 본인 commit 수집
- 문서 업로드 및 텍스트 추출

### 6.3 맞춤 자소서 생성
- 회사명, 직무, 지원 유형, 문항 입력
- 직무별 기본 템플릿 기반 자소서 초안 생성
- 동일 포트폴리오로 복수 지원 버전 저장

### 6.4 면접 질문 생성
- 자소서와 포트폴리오 기반 예상 질문/기술 질문/꼬리 질문 생성
- 질문 유형과 난이도 저장

### 6.5 텍스트 기반 모의 면접
- 질문 세트를 기반으로 세션 생성
- 답변 저장, 건너뛰기, 자동 일시정지, 재개 지원

### 6.6 면접 결과 저장 및 히스토리 조회
- 세션 종료 후 결과 리포트, 태그, 코멘트 저장
- 과거 세션과 결과 히스토리 조회

## 10. 기술 선택 이유

### 10.1 Spring Boot
- 인증, API, 검증, 예외 처리, 테스트 생태계를 한 프레임에서 안정적으로 구성할 수 있다.

### 10.2 PostgreSQL
- 사용자/포트폴리오/application/세션 관계를 정합성 있게 관리하기 좋다.

### 10.3 Redis
- 세션/캐시/외부 연동 보조 처리에 유리하다.

### 10.4 GitHub API
- 개발자 포트폴리오의 핵심 근거 데이터를 구조화해서 수집할 수 있다.

### 10.5 AI API 추상화
- 자소서 생성, 질문 생성, 평가를 템플릿 기반으로 통제하고 재시도/검증을 분리하기 좋다.

## 11. 시스템 설계 방향

### 11.1 도메인 구조
- Auth / User
- GitHub Portfolio
- Document Upload / Extraction
- Application
- AI Generation
- Interview Session / Result

### 11.2 설계 원칙
- 도메인 우선 구조를 사용한다.
- `application` 중심 흐름을 유지한다.
- 로그인 계정과 GitHub 연동을 분리한다.
- 질문 세트와 실제 세션을 분리한다.
- 문서에 없는 상태값, enum, 오류 코드를 임의로 추가하지 않는다.

## 12. 협업 방식

### 12.1 역할 분담
- 플랫폼/공통 기반 + 인프라/배포/관측성
- 포트폴리오 수집/외부 연동 + 품질/테스트
- AI 기능/생성 파이프라인
- 면접 세션/서비스 흐름 + 대시보드/히스토리

### 12.2 공통 협업 원칙
- 문서가 코드보다 먼저다.
- 같은 변경은 같은 PR에서 닫는다.
- API, DB, 화면 상태, AI 출력 스키마가 함께 바뀌면 관련 문서를 같이 수정한다.

### 12.3 개발 프로세스
- 요구사항, OpenAPI, DB, 화면 정의, WBS를 먼저 잠근다.
- 그 다음 공통 기반, 포트폴리오 수집, 지원 준비/생성, 면접 흐름, 품질 보강 순으로 구현한다.

## 13. 서비스 차별점

### 13.1 단순 채팅형 AI와의 차이
- 단발성 프롬프트가 아니라 포트폴리오를 저장하고 재사용한다.

### 13.2 개발자 취업 준비 흐름 특화
- GitHub 활동과 문서형 포트폴리오를 동시에 활용한다.

### 13.3 결과 기록 기반 학습
- 세션 결과를 누적해 개인별 약점을 추적할 수 있다.
