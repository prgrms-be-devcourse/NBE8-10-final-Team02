# AI 기술 면접 연습 플랫폼 요구사항 명세서

> 현재까지 확정된 핵심 기능 요구사항을 반영한 v0.2 초안

## 1. 문서 개요

- **작성 목적**: 서비스의 범위, 사용자 흐름, 기능 요구사항을 현재 확정 수준까지 명확히 정리하여 이후 ERD, API, 와이어프레임, WBS 작성의 기준으로 사용한다.

- **적용 범위**: 1차 MVP를 중심으로 한 요구사항 정의

- **문서 기준**: 기능 정의서, 프로젝트 아이디어 정리본, 역할 분담안을 바탕으로 현재까지 합의된 내용을 반영한다.

## 2. 서비스 개요

본 서비스는 개발자 취업 준비생이 자신의 GitHub 활동, 이력서, 수상기록 등 포트폴리오 데이터를 한 번 수집하고 저장한 뒤, 이를 바탕으로 맞춤 자소서 생성, 예상 면접 질문 생성, 텍스트 기반 모의 면접, 결과 히스토리 관리까지 하나의 흐름으로 수행할 수 있도록 지원하는 AI 기반 취업 준비 플랫폼이다.

기존 생성형 AI 사용 방식은 사용자가 매번 자료를 다시 입력해야 하고, 자소서 작성, 공고 분석, 면접 연습이 분리되어 있는 경우가 많다. 본 서비스는 포트폴리오 데이터를 재사용 가능한 자산으로 관리함으로써 반복 입력 부담을 줄이고, 취업 준비 전 과정을 하나의 서비스 안에서 연결하는 것을 목표로 한다.

## 3. 범위 정의

### 3.1 MVP 포함 범위

- OAuth2 로그인
- GitHub 연동 및 repository 목록 조회
- 사용자 본인 commit 수집
- 이력서 및 기타 증빙 문서 업로드
- 문서 텍스트 추출 및 포트폴리오 데이터 저장
- 직무별 템플릿 기반 맞춤 자소서 생성
- 포트폴리오 기반 면접 질문 생성
- 텍스트 기반 모의 면접
- 면접 결과 저장 및 히스토리 조회

### 3.2 MVP 제외 범위

- PR 및 Issue 수집
- 음성 및 영상 기반 모의 면접
- 결제, 광고, 토큰 시스템
- 대규모 통계 및 커뮤니티 기능
- 게임화 보상 시스템

## 4. 사용자 유형

- 부트캠프 수료 후 취업 준비 중인 개발자
- 신입 개발자 취업 준비생
- 이직을 준비 중인 주니어 개발자
- 자신의 GitHub, 프로젝트 경험, 이력서를 기반으로 맞춤형 취업 준비를 하고 싶은 사용자

## 5. 전체 사용자 흐름

1. 사용자는 GitHub, Google, Kakao 중 하나를 선택해 로그인한다.
1. 사용자는 GitHub 계정을 연동하거나 GitHub URL을 입력하고, 이력서 및 기타 증빙 문서를 업로드한다.
1. 시스템은 repository 목록과 사용자 본인 commit 내역을 수집하고, 문서 텍스트를 추출해 포트폴리오 데이터를 저장한다.
1. 사용자는 지원 직무, 회사명, 자소서 문항을 입력해 맞춤 자소서 초안을 생성한다.
1. 사용자는 생성된 자소서와 포트폴리오를 바탕으로 면접 질문을 생성하고, 텍스트 기반 모의 면접을 진행할 수 있다.
1. 시스템은 면접 세션 종료 후 결과를 저장하고, 히스토리와 피드백 데이터를 조회할 수 있도록 제공한다.

## 6. 확정 정책

- 로그인 범위는 GitHub, Google, Kakao OAuth2 로그인을 지원한다.
- GitHub 연동 범위는 사용자 repository 목록 조회와 사용자 본인 commit 수집까지를 MVP 범위로 한다.
- PR 및 Issue 수집은 후속 확장 기능으로 분리한다.
- 자소서 생성은 직무별 기본 템플릿을 우선 적용한다.
- 회사별 또는 공고별 자소서 문항은 사용자가 직접 입력할 수 있다.
- 면접 결과는 세션 총점, 질문별 점수, 약점 태그, 간단한 평가 근거를 저장한다.

## 7. 기능 요구사항

### 7.1 FR-01 회원가입 및 인증

- **목적**: 사용자가 서비스에 안전하게 가입하고 로그인할 수 있도록 하며, 이후 포트폴리오 등록 및 개인화 기능 사용을 위한 사용자 식별 기반을 제공한다.

- **사용자**: 비회원 사용자 / 기존 회원 사용자

**선행조건**

- 사용자는 유효한 GitHub, Google 또는 Kakao 계정을 보유해야 한다.
- 외부 OAuth2 제공자 연동이 정상 동작해야 한다.

**입력**

- 로그인 제공자 선택 - GitHub / Google / Kakao
- OAuth2 인증 및 동의 결과

**처리 흐름**

1. 사용자는 로그인 화면에서 로그인 제공자를 선택한다.
1. 시스템은 선택한 OAuth2 제공자의 인증 화면으로 사용자를 이동시킨다.
1. 사용자가 인증 및 동의를 완료하면 시스템은 인증 결과를 수신한다.
1. 시스템은 제공자로부터 받은 사용자 식별 정보를 기준으로 기존 회원 여부를 확인한다.
1. 기존 회원이면 로그인 처리하고, 신규 사용자이면 회원 정보를 생성한 후 로그인 처리한다.
1. 로그인 성공 시 메인 화면 또는 온보딩 화면으로 이동한다.

**출력**

- 로그인 성공 여부
- 사용자 기본 정보
- 세션 또는 토큰 발급 결과
- 신규 사용자 여부

**Validation**

- 지원하지 않는 로그인 제공자는 선택할 수 없어야 한다.
- OAuth2 인증 결과에 사용자 식별 정보가 없으면 로그인 처리할 수 없어야 한다.
- 동일한 제공자 계정으로 중복 회원 생성이 발생하면 안 된다.

**예외 케이스**

- 사용자가 외부 인증 도중 취소한 경우
- OAuth2 제공자 서버 장애 또는 응답 지연
- 인증은 완료되었지만 필수 사용자 정보가 누락된 경우
- 이미 탈퇴 처리된 계정으로 재로그인을 시도한 경우

**오류 메시지 예시**

- 로그인이 취소되었습니다.
- 로그인 처리 중 오류가 발생했습니다. 다시 시도해주세요.
- 지원하지 않는 로그인 방식입니다.
- 필수 사용자 정보를 가져오지 못했습니다.

**수용 기준**

- 사용자는 GitHub, Google, Kakao 중 하나를 통해 로그인할 수 있어야 한다.
- 신규 사용자는 최초 로그인 시 회원 생성 후 즉시 서비스 진입이 가능해야 한다.
- 기존 사용자는 동일 계정으로 재로그인 시 기존 계정으로 연결되어야 한다.

### 7.2 FR-02 포트폴리오 수집 및 저장

- **목적**: 사용자의 GitHub 활동과 업로드 문서를 수집, 정규화, 저장하여 자소서 생성과 면접 기능의 기반 데이터로 활용한다.

- **사용자**: 로그인한 회원 사용자

**선행조건**

- 사용자는 로그인 상태여야 한다.
- GitHub 연동 또는 GitHub URL 입력이 가능해야 한다.
- 파일 업로드 기능이 정상 동작해야 한다.

**입력**

- GitHub 계정 연동 정보 또는 GitHub URL
- 이력서 파일
- 수상기록 및 기타 증빙 파일
- 파일 메타데이터 - 파일명 / 파일 형식 / 파일 크기

**처리 흐름**

1. 사용자는 포트폴리오 등록 화면에서 GitHub 계정을 연동하거나 GitHub URL을 입력한다.
1. 시스템은 사용자 repository 목록을 조회한다.
1. 시스템은 사용자가 선택한 repository에 대해 사용자 본인 commit 내역을 수집한다.
1. 사용자는 이력서 및 기타 증빙 파일을 업로드한다.
1. 시스템은 업로드된 파일 형식과 용량을 검증한다.
1. 시스템은 PDF 또는 지원 문서에서 텍스트를 추출한다.
1. 시스템은 수집한 GitHub 데이터와 문서 텍스트를 정규화하여 저장한다.
1. 저장 완료 후 포트폴리오 등록 결과를 사용자에게 보여준다.

**출력**

- repository 목록
- commit 수집 결과
- 업로드 파일 저장 결과
- 텍스트 추출 결과
- 포트폴리오 저장 완료 상태

**Validation**

- GitHub URL은 유효한 형식이어야 한다.
- repository 조회가 가능한 계정 또는 URL이어야 한다.
- 지원 파일 형식만 업로드 가능해야 한다.
- 허용 용량을 초과한 파일은 업로드할 수 없어야 한다.
- 로그인 사용자와 수집 대상 데이터의 연결 관계가 저장되어야 한다.

**예외 케이스**

- GitHub 인증 실패 또는 권한 부족
- repository가 존재하지 않거나 비공개 접근 권한이 없는 경우
- commit 데이터가 없거나 사용자 본인 commit을 식별할 수 없는 경우
- 파일 업로드 실패
- 텍스트 추출 실패
- 일부 데이터만 저장되고 나머지가 실패한 경우

**오류 메시지 예시**

- GitHub 정보를 불러오지 못했습니다.
- 접근 권한이 없는 repository입니다.
- 업로드할 수 없는 파일 형식입니다.
- 파일 용량이 허용 범위를 초과했습니다.
- 문서 내용 추출에 실패했습니다.
- 포트폴리오 저장 중 일부 항목 처리에 실패했습니다.

**수용 기준**

- 사용자는 GitHub 계정 연동 또는 URL 입력을 통해 repository 목록을 조회할 수 있어야 한다.
- 사용자는 선택한 repository 기준으로 본인 commit 내역을 수집할 수 있어야 한다.
- 사용자는 이력서와 기타 증빙 파일을 업로드할 수 있어야 한다.
- 시스템은 업로드 문서에서 텍스트를 추출하여 저장할 수 있어야 한다.
- 저장된 포트폴리오 데이터는 이후 자소서 생성 및 면접 기능에서 재사용 가능해야 한다.

### 7.3 FR-03 맞춤 자소서 생성

- **목적**: 저장된 포트폴리오 데이터와 사용자가 입력한 지원 정보에 기반하여 맞춤형 자소서 초안을 생성한다.

- **사용자**: 로그인한 회원 사용자

**선행조건**

- 사용자의 포트폴리오 데이터가 1건 이상 저장되어 있어야 한다.
- 사용자가 지원 직무를 선택하거나 입력해야 한다.

**입력**

- 지원 직무
- 회사명
- 자소서 문항
- 사용할 포트폴리오 데이터 선택값
- 생성 옵션 - 문체 / 분량 / 강조 포인트

**처리 흐름**

1. 사용자는 자소서 생성 화면에서 지원 직무를 입력 또는 선택한다.
1. 시스템은 직무별 기본 템플릿을 적용한다.
1. 사용자는 회사명과 자소서 문항을 입력할 수 있다.
1. 시스템은 선택된 포트폴리오 데이터와 입력 문항을 기반으로 자소서 초안을 생성한다.
1. 시스템은 생성 결과를 사용자에게 보여준다.
1. 사용자는 초안을 수정하거나 저장할 수 있다.

**출력**

- 자소서 초안
- 문항별 생성 결과
- 저장 여부

**Validation**

- 지원 직무는 필수 입력값이어야 한다.
- 자소서 문항이 비어 있으면 생성할 수 없어야 한다.
- 포트폴리오 데이터가 없는 사용자는 생성 기능을 사용할 수 없어야 한다.

**예외 케이스**

- 포트폴리오 데이터 부족으로 생성 품질이 낮은 경우
- AI 생성 응답 실패
- 입력 문항 형식이 비정상적인 경우

**오류 메시지 예시**

- 지원 직무를 입력해주세요.
- 자소서 문항을 입력해주세요.
- 포트폴리오 데이터가 부족하여 생성할 수 없습니다.
- 자소서 생성 중 오류가 발생했습니다.

**수용 기준**

- 사용자는 직무별 템플릿을 기반으로 자소서 초안을 생성할 수 있어야 한다.
- 사용자는 회사명과 문항을 직접 입력하여 생성 결과를 조정할 수 있어야 한다.
- 생성 결과는 저장 및 재수정이 가능해야 한다.

### 7.4 FR-04 면접 질문 생성

- **목적**: 저장된 포트폴리오 데이터와 자소서 초안을 바탕으로 예상 면접 질문을 생성하여 사용자의 사전 연습을 지원한다.

- **사용자**: 로그인한 회원 사용자

**선행조건**

- 사용자의 포트폴리오 데이터가 저장되어 있어야 한다.
- 자소서 초안이 존재하거나 면접 질문 생성에 필요한 기본 입력값이 제공되어야 한다.

**입력**

- 지원 직무
- 회사명
- 자소서 초안 또는 선택된 포트폴리오 데이터
- 질문 생성 옵션 - 질문 개수 / 난이도 / 질문 유형

**처리 흐름**

1. 사용자는 면접 질문 생성 화면에서 지원 직무와 회사명 등 기본 정보를 입력한다.
1. 시스템은 저장된 포트폴리오 데이터와 자소서 초안을 분석한다.
1. 시스템은 기술 질문, 경험 기반 질문, 꼬리 질문을 포함한 예상 면접 질문 목록을 생성한다.
1. 시스템은 생성된 질문을 사용자에게 보여주고, 사용자는 필요한 질문을 선택하거나 저장할 수 있다.

**출력**

- 질문 목록
- 질문 유형 정보
- 질문 저장 여부

**Validation**

- 지원 직무는 필수 입력값이어야 한다.
- 질문 개수와 난이도는 허용된 범위 안에서만 선택 가능해야 한다.
- 포트폴리오 또는 자소서 기반 데이터가 부족한 경우 생성 품질 저하 안내가 가능해야 한다.

**예외 케이스**

- 포트폴리오 데이터가 부족하여 질문 품질이 낮은 경우
- AI 질문 생성 응답 실패
- 지원 직무 또는 입력 정보가 누락된 경우

**오류 메시지 예시**

- 지원 직무를 입력해주세요.
- 면접 질문 생성에 필요한 데이터가 부족합니다.
- 면접 질문 생성 중 오류가 발생했습니다.

**수용 기준**

- 사용자는 포트폴리오 및 자소서 정보를 바탕으로 예상 면접 질문을 생성할 수 있어야 한다.
- 생성된 질문은 기술 질문과 경험 기반 질문을 포함할 수 있어야 한다.
- 사용자는 생성된 질문을 확인하고 이후 모의 면접에 활용할 수 있어야 한다.

### 7.5 FR-05 텍스트 기반 모의 면접

- **목적**: 생성된 면접 질문을 바탕으로 사용자가 텍스트 기반으로 면접 연습을 진행하고 답변을 제출할 수 있도록 지원한다.

- **사용자**: 로그인한 회원 사용자

**선행조건**

- 면접 질문이 1개 이상 준비되어 있어야 한다.
- 사용자가 로그인 상태여야 한다.

**입력**

- 선택한 질문 세트
- 질문별 사용자 답변 텍스트
- 세션 진행 옵션 - 질문 수 / 난이도 / 종료 여부

**처리 흐름**

1. 사용자는 생성된 면접 질문 목록 또는 저장된 질문 세트를 선택해 모의 면접을 시작한다.
1. 시스템은 질문을 순차적으로 제시하고, 사용자는 각 질문에 대해 텍스트 답변을 입력한다.
1. 시스템은 필요 시 이전 답변을 바탕으로 후속 질문 또는 꼬리 질문을 생성할 수 있다.
1. 사용자가 세션을 종료하면 시스템은 답변 데이터를 평가 단계로 전달한다.

**출력**

- 현재 질문 정보
- 질문별 사용자 답변
- 세션 진행 상태 또는 종료 상태

**Validation**

- 질문 세트가 없으면 모의 면접을 시작할 수 없어야 한다.
- 빈 답변은 제출할 수 없으며, 건너뛰기는 별도 동작으로 구분되어야 한다.
- 세션 종료 전에는 결과 저장이 완료 상태로 처리되면 안 된다.

**예외 케이스**

- 면접 세션 도중 사용자가 종료한 경우
- 질문 생성 또는 꼬리 질문 생성 응답 실패
- 답변 저장 또는 세션 종료 처리 실패

**오류 메시지 예시**

- 진행 가능한 면접 질문이 없습니다.
- 답변을 입력해주세요.
- 모의 면접 진행 중 오류가 발생했습니다.

**수용 기준**

- 사용자는 질문을 순차적으로 확인하며 텍스트 답변을 입력할 수 있어야 한다.
- 모의 면접은 질문 단위로 진행되며 세션 종료가 가능해야 한다.
- 종료된 세션의 답변 데이터는 결과 평가 및 저장 단계로 전달되어야 한다.

### 7.6 FR-06 면접 결과 저장 및 히스토리 조회

- **목적**: 모의 면접 세션 결과를 저장하고, 사용자가 이전 면접 기록과 피드백을 다시 확인할 수 있도록 지원한다.

- **사용자**: 로그인한 회원 사용자

**선행조건**

- 면접 세션이 종료되어야 한다.
- 질문별 답변 데이터가 저장되어 있어야 한다.

**입력**

- 종료된 면접 세션 데이터
- 질문별 답변 텍스트
- 평가 결과 데이터 - 점수 / 약점 태그 / 간단한 평가 근거

**처리 흐름**

1. 시스템은 종료된 면접 세션의 질문과 답변을 바탕으로 평가 결과를 생성한다.
1. 시스템은 세션 총점, 질문별 점수, 약점 태그, 간단한 평가 근거를 저장한다.
1. 사용자는 결과 화면에서 이번 세션의 요약 정보를 확인할 수 있다.
1. 사용자는 히스토리 화면에서 이전 세션 목록과 상세 결과를 조회할 수 있다.

**출력**

- 세션 총점
- 질문별 점수
- 약점 태그
- 간단한 평가 근거
- 세션별 히스토리 목록

**Validation**

- 종료되지 않은 세션은 최종 결과로 저장할 수 없어야 한다.
- 점수는 정의된 범위 안에서 저장되어야 한다.
- 질문별 결과는 세션과 사용자 계정에 올바르게 연결되어야 한다.

**예외 케이스**

- 평가 결과 생성 실패
- 결과 저장 실패 또는 일부 항목 저장 실패
- 히스토리 조회 대상 데이터가 존재하지 않는 경우

**오류 메시지 예시**

- 면접 결과 생성 중 오류가 발생했습니다.
- 면접 결과 저장에 실패했습니다.
- 조회 가능한 면접 기록이 없습니다.

**수용 기준**

- 종료된 면접 세션은 세션 총점, 질문별 점수, 약점 태그, 간단한 평가 근거와 함께 저장되어야 한다.
- 사용자는 자신의 면접 히스토리를 최신순으로 조회할 수 있어야 한다.
- 사용자는 특정 세션의 상세 결과를 다시 확인할 수 있어야 한다.
