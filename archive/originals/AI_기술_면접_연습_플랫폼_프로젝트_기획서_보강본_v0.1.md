# AI 기술 면접 연습 플랫폼 프로젝트 기획서

> 제출용 보강본 v0.1

## 1. 프로젝트 개요

### 1.1 프로젝트명
AI 기술 면접 연습 플랫폼

### 1.2 한 줄 소개
개발자 취업 준비생이 자신의 GitHub 활동, 이력서, 수상기록 등 포트폴리오를 기반으로 자소서 작성부터 예상 면접 질문 생성, 텍스트 기반 모의 면접, 결과 히스토리 관리까지 한 흐름으로 수행할 수 있도록 지원하는 AI 기반 취업 준비 서비스

### 1.3 프로젝트 배경
최근 생성형 AI를 활용한 자기소개서 작성과 면접 준비 수요가 증가하고 있지만, 대부분의 사용 방식은 단발성 채팅에 머물러 있다. 사용자는 GitHub 활동, 프로젝트 경험, 이력서, 지원 직무 정보를 매번 다시 입력해야 하며, 자소서 작성과 면접 준비 또한 서로 분리된 도구를 사용해야 하는 불편이 있다.

특히 개발자 취업 준비의 경우, 단순 문장 생성보다 GitHub 커밋 이력, 프로젝트 경험, 기술 선택 이유, 문제 해결 과정과 같은 포트폴리오 기반 정보가 더 중요하다. 그러나 기존 서비스들은 이러한 데이터를 구조화하여 재사용하기보다, 일회성 입력에 의존하는 경우가 많다.

본 프로젝트는 이러한 문제를 해결하기 위해 사용자의 포트폴리오를 한 번 수집하고 저장한 뒤, 이를 자소서 생성과 면접 질문 생성, 모의 면접까지 재사용 가능한 데이터 자산으로 활용하는 서비스를 목표로 한다.

## 2. 문제 정의

### 2.1 기존 방식의 한계
- 사용자가 GitHub, 이력서, 수상기록, 지원 정보를 매번 반복 입력해야 한다.
- 자소서 작성, 예상 질문 생성, 모의 면접이 각각 분리되어 있어 준비 흐름이 끊긴다.
- 일반적인 AI 채팅은 이전 대화 맥락이 약하거나 장기적으로 관리되지 않는다.
- 개발자 포트폴리오에서 중요한 저장소 활동, 커밋 경험, 프로젝트 근거가 체계적으로 관리되지 않는다.
- 면접 연습 결과가 기록되지 않아 반복 학습과 개선 포인트 추적이 어렵다.

### 2.2 해결하고자 하는 핵심 문제
- 포트폴리오 기반 정보를 한 번 수집하여 지속적으로 재사용할 수 있도록 한다.
- 자소서 작성과 면접 준비를 하나의 연결된 사용자 흐름으로 통합한다.
- 개발자 취업 준비에 맞는 GitHub 기반 데이터 활용 구조를 제공한다.
- 면접 결과를 점수, 태그, 근거와 함께 저장하여 반복 학습 기반을 만든다.

## 3. 프로젝트 목표

### 3.1 기능적 목표
- GitHub, 이력서, 수상기록 등 포트폴리오 데이터를 수집하고 저장한다.
- 저장된 데이터를 기반으로 직무 중심의 맞춤 자소서 초안을 생성한다.
- 자소서와 포트폴리오를 바탕으로 예상 면접 질문을 생성한다.
- 텍스트 기반 모의 면접을 통해 질문-답변 흐름을 연습할 수 있도록 한다.
- 면접 결과를 저장하고 히스토리 형태로 다시 확인할 수 있도록 한다.

### 3.2 서비스적 목표
- 개발자 취업 준비 과정을 하나의 서비스에서 연결한다.
- 단순 AI 생성 결과 제공이 아니라, 준비 과정 전체를 관리하는 경험을 제공한다.
- 향후 유료 플랜, 고성능/저비용 모델 분기, 통계 기능 등으로 확장 가능한 구조를 만든다.

## 4. 타겟 사용자

- 부트캠프 수료 후 취업 준비 중인 개발자
- 신입 개발자 취업 준비생
- 이직을 준비 중인 주니어 개발자
- 자신의 GitHub, 프로젝트 경험, 이력서를 바탕으로 맞춤형 자소서와 면접 준비를 하고 싶은 사용자

## 5. 서비스 핵심 가치

### 5.1 포트폴리오 재사용 구조
사용자가 GitHub 저장소, 커밋, 이력서, 수상기록 등을 한 번 입력하면 이후 자소서 생성과 면접 준비에 반복 활용할 수 있다.

### 5.2 자소서와 면접 준비의 연결
자소서 생성 결과를 다시 면접 질문 생성과 모의 면접 세션에 활용함으로써, 실제 취업 준비 흐름과 유사한 사용자 경험을 제공한다.

### 5.3 개발자 특화 데이터 활용
일반적인 이력 정보뿐 아니라 GitHub 활동과 프로젝트 기록을 활용하여, 개발자 취업 준비 서비스에 더 적합한 입력 구조를 제공한다.

### 5.4 결과 기록과 반복 학습
모의 면접 결과를 점수, 약점 태그, 간단한 평가 근거와 함께 저장하여 사용자가 스스로 약점을 추적하고 반복 학습할 수 있도록 한다.

## 6. 주요 기능

### 6.1 회원가입 및 인증
- GitHub, Google, Kakao OAuth2 로그인 지원
- 사용자 식별 및 개인화 기능의 기반 제공

### 6.2 포트폴리오 수집 및 저장
- GitHub 계정 연동 또는 URL 입력
- repository 목록 조회
- 사용자 본인 commit 수집
- 이력서 및 수상기록 파일 업로드
- 문서 텍스트 추출 및 데이터 저장

### 6.3 맞춤 자소서 생성
- 지원 직무 기반 템플릿 적용
- 회사명 및 자소서 문항 직접 입력
- 포트폴리오 기반 강점 반영
- 생성 결과 저장 및 재조회

### 6.4 면접 질문 생성
- 자소서 기반 예상 질문 생성
- 프로젝트 경험 기반 질문 생성
- 기술 질문 및 꼬리 질문 생성
- 질문 유형 분류 및 저장

### 6.5 텍스트 기반 모의 면접
- 질문 세트 선택 및 세션 시작
- 질문 순차 진행
- 답변 입력 및 저장
- 세션 종료 처리

### 6.6 면접 결과 저장 및 히스토리 조회
- 세션 총점 저장
- 질문별 점수 저장
- 약점 태그 및 간단한 근거 저장
- 이전 세션 히스토리 재조회

## 7. 전체 사용자 흐름

1. 사용자는 GitHub, Google, Kakao 중 하나를 통해 로그인한다.
2. 사용자는 GitHub를 연동하거나 GitHub URL을 입력하고, 이력서 및 기타 증빙 문서를 업로드한다.
3. 시스템은 repository 목록과 사용자 본인 commit 내역, 문서 텍스트를 수집하여 포트폴리오 데이터로 저장한다.
4. 사용자는 지원 직무, 회사명, 자소서 문항을 입력하고 자소서 초안을 생성한다.
5. 시스템은 저장된 포트폴리오와 자소서 초안을 바탕으로 예상 면접 질문을 생성한다.
6. 사용자는 질문 세트를 선택해 텍스트 기반 모의 면접을 진행한다.
7. 시스템은 세션 종료 후 결과를 저장하고, 히스토리와 피드백을 제공한다.

## 10. 기술 선택 이유

### 10.1 Spring Boot
백엔드 중심 팀 프로젝트에서 인증, API 설계, 외부 연동, 예외 처리, 테스트 구조를 안정적으로 구성하기에 적합하다.

### 10.2 PostgreSQL
관계형 데이터 모델이 명확하고, 사용자-포트폴리오-자소서-면접 기록 간 관계를 안정적으로 관리하기에 적합하다.

### 10.3 Redis
세션 캐시, 임시 상태 저장, 추후 비동기 작업 큐 또는 rate limit 대응에 확장 여지가 있다.

### 10.4 GitHub API
개발자 취업 준비 서비스의 핵심 차별점인 GitHub 기반 포트폴리오 활용을 위해 필수적이다.

### 10.5 AI API 추상화
하나의 AI 모델에 종속되지 않도록 추상화 계층을 둠으로써 비용, 품질, fallback 전략을 유연하게 구성할 수 있다.

## 12. 협업 방식

### 12.1 역할 분담
1. 플랫폼/공통 기반 + 인프라/배포/관측성 담당
2. 포트폴리오 수집/외부 연동 + 품질/테스트 담당
3. AI 기능/생성 파이프라인 담당
4. 면접 세션/서비스 흐름 + 대시보드/히스토리 담당

### 12.2 공통 협업 원칙
- 기능 개발 이전에 요구사항, ERD, 공통 개발 규칙을 문서로 확정한다.
- 공통 응답 형식, 예외 처리, API 규칙은 초반에 통일한다.
- GitHub Actions 기반 CI를 통해 최소한의 품질 검증을 자동화한다.
- 보안, 성능, 비용은 특정 한 명이 아니라 각 도메인 담당자가 함께 책임진다.

### 12.3 개발 프로세스
- 기획 확정
- 요구사항 명세서 작성
- ERD 및 DB 설계
- API 명세 작성
- 구현
- 통합 테스트 및 문서 보완

## 13. 서비스 차별점

### 13.1 단순 채팅형 AI와의 차이
일회성 프롬프트 입력이 아니라 포트폴리오를 저장하고 재사용한다는 점에서 차이가 있다.

### 13.2 개발자 취업 준비 흐름 특화
GitHub 활동을 기반으로 자소서와 면접 준비를 연결하므로, 일반적인 자기소개서 생성 서비스보다 개발자 도메인에 더 적합하다.

### 13.3 결과 기록 기반 학습
면접 결과를 축적하고 약점 태그로 관리함으로써, 단순 생성 서비스가 아니라 학습형 서비스로 확장할 수 있다.

## 15. 리스크 및 대응 전략

### 15.1 GitHub API 제한
문제
- rate limit, pagination, 동기화 정책 문제 발생 가능

대응
- MVP에서는 본인 commit 수집까지만 범위를 제한한다.
- 동기화 시점과 재시도 전략을 별도로 설계한다.

### 15.2 PDF 텍스트 추출 품질
문제
- 문서 형식에 따라 추출 정확도가 달라질 수 있다.

대응
- 지원 형식을 제한하고, 추출 실패 상태를 저장한다.
- 원문 파일과 추출 텍스트를 분리 관리한다.

### 15.3 AI 비용 및 품질 편차
문제
- 생성 비용이 높아질 수 있고, 결과 품질이 일정하지 않을 수 있다.

대응
- 모델 추상화 계층을 두고 fallback 전략을 준비한다.
- 템플릿 기반 생성과 결과 저장 구조를 통해 재호출을 줄인다.

### 15.4 개인정보 및 포트폴리오 보안
문제
- 이력서, 수상기록, GitHub 활동 정보는 민감할 수 있다.

대응
- 최소한의 데이터만 저장하고 접근 권한을 사용자 기준으로 제한한다.
- 로그 및 예외 메시지에 민감한 내용을 직접 남기지 않는다.
