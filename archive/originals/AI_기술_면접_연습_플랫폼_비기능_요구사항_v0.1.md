# AI 기술 면접 연습 플랫폼 비기능 요구사항

## 1. 문서 개요

- 문서명: AI 기술 면접 연습 플랫폼 비기능 요구사항
- 버전: v0.1
- 작성 목적: 현재 확정된 MVP 기능을 실제 서비스 수준으로 구현하고 운영하기 위해 필요한 성능, 보안, 안정성, 운영, 테스트 기준을 정의한다.
- 적용 범위: 1차 MVP 기준 백엔드/API/DB/외부 연동/운영 환경 전반
- 기준 문서:
  - AI 기술 면접 연습 플랫폼 요구사항 명세서 v0.2
  - AI 기술 면접 연습 플랫폼 ERD 초안 v0.1
  - AI 기술 면접 연습 플랫폼 DB 설계 정리 v0.1

## 2. 작성 원칙

- 본 문서는 기능 요구사항을 대체하지 않고, 기능 요구사항을 안정적으로 구현하기 위한 품질 기준을 정의한다.
- 수치 기준이 아직 확정되지 않은 항목은 `초안 제안`으로 명시한다.
- MVP 범위를 넘어가는 항목은 `후속 확장` 또는 `미결정 항목`으로 분리한다.
- 기능 범위와 충돌하는 경우 요구사항 명세서의 범위를 우선한다.

## 3. 비기능 요구사항 범위

본 문서는 아래 영역을 다룬다.

- 성능 및 응답 시간
- 가용성 및 장애 대응
- 외부 연동 신뢰성

- AI 생성 기능 운영 기준

## 4. 비기능 요구사항 상세

### 4.1 보안 및 인증

#### 4.1.1 인증 정책

- 모든 사용자 전용 API는 인증된 사용자만 접근할 수 있어야 한다.
- OAuth2 로그인은 GitHub, Google, Kakao만 허용한다.
- 인증되지 않은 요청은 보호된 리소스에 접근할 수 없어야 한다.
- 탈퇴 또는 비활성 상태 사용자는 로그인 또는 주요 기능 접근이 제한되어야 한다.

#### 4.1.2 인가 정책

- 사용자는 자신의 문서, GitHub 연결, application, 면접 세션, 결과 데이터에만 접근할 수 있어야 한다.
- 다른 사용자의 resource ID를 직접 입력하더라도 데이터 조회 또는 수정이 가능하면 안 된다.
- 삭제, 수정, 결과 조회 API는 소유권 검증을 반드시 수행해야 한다.

#### 4.1.3 민감정보 보호

- access token, OAuth credential, 외부 API secret은 평문으로 로그에 남기면 안 된다.
- GitHub access token 등 외부 연동 credential을 저장해야 하는 경우 암호화 저장 또는 이에 준하는 보호 조치를 적용해야 한다.
- 운영 환경에서는 HTTPS/TLS를 기본으로 사용해야 한다.
- 환경 변수, secret, API key는 코드 저장소에 커밋되면 안 된다.

#### 4.1.4 보안 실패 처리

- 인증 실패, 권한 부족, 지원하지 않는 제공자 선택 등은 구분된 오류 코드와 메시지로 응답해야 한다.
- 반복적인 로그인 실패 또는 비정상 요청 패턴에 대해서는 rate limit 또는 차단 정책을 둘 수 있어야 한다.

### 4.2 개인정보 및 데이터 보호

#### 4.2.1 수집 최소화

- 서비스 제공에 필요한 최소 범위의 사용자 정보만 저장해야 한다.
- OAuth 제공자별로 이메일이 없을 수 있으므로 이메일은 선택 저장을 허용한다.
- 생성형 AI 처리에 불필요한 개인정보는 프롬프트 컨텍스트에 포함하지 않는 것을 원칙으로 한다.

#### 4.2.2 사용자 데이터 분리

- 사용자별 포트폴리오 문서, GitHub 데이터, 자소서, 면접 결과는 논리적으로 분리되어야 한다.
- 다른 사용자 데이터가 AI 생성 컨텍스트에 섞이면 안 된다.
- 동일한 질문 세트나 생성 기능을 사용하더라도 사용자 간 저장 결과는 독립적으로 관리되어야 한다.

#### 4.2.3 로그 마스킹

- 이메일, 토큰, 원문 문서 전체 내용, 사용자 답변 전체 원문은 운영 로그에 그대로 기록하지 않는 것을 기본으로 한다.
- 로그가 필요한 경우 일부 마스킹 또는 요약 형태로 남겨야 한다.

#### 4.2.4 삭제 및 보관 정책

- 사용자 삭제 시 하위 데이터 삭제 정책은 DB 설계 문서를 따른다.
- 참조 중인 documents, github_repositories, feedback_tags는 기본적으로 삭제를 제한한다.
- 개인정보 보관 기간과 로그 보관 기간은 운영 정책으로 별도 정해야 한다.

### 4.3 성능 및 응답 시간

#### 4.3.1 기본 원칙

- 외부 연동이 없는 단순 조회 API는 빠르게 응답해야 한다.
- 시간이 오래 걸릴 수 있는 동기화/추출/생성 작업은 사용자에게 현재 상태를 명확히 전달해야 한다.
- 성능 기준은 p95를 기본 측정 기준으로 삼는다.

#### 4.3.2 응답 시간 목표 초안

- 로그인 후 내 정보 조회: p95 1초 이내
- 일반 목록/상세 조회 API: p95 1초 이내
- application 생성/수정/삭제: p95 2초 이내
- 면접 답변 저장: p95 2초 이내
- 면접 결과 상세 조회: p95 2초 이내
- GitHub repository 목록 조회: p95 3초 이내
- 문서 업로드 초기 응답: p95 5초 이내
- 문서 텍스트 추출 완료: 일반적인 10페이지 내 문서 기준 60초 이내 목표
- 자소서 초안 생성: p95 30초 이내
- 면접 질문 생성: p95 20초 이내
- 면접 결과 생성: p95 30초 이내

#### 4.3.3 성능 초과 시 처리

- 기준 시간을 초과할 가능성이 높은 작업은 즉시 실패시키기보다 진행 상태를 저장하고 재조회 가능해야 한다.
- 성능 기준을 반복적으로 초과하는 API는 로그와 메트릭으로 식별 가능해야 한다.

### 4.4 가용성 및 안정성

#### 4.4.1 서비스 가용성 목표

- MVP 운영 환경의 월간 가용성 목표는 99.0% 이상을 초안 기준으로 한다.
- 계획된 점검은 사전에 공지하거나 내부 운영 시간으로 제한한다.

#### 4.4.2 장애 허용 원칙

- 일부 외부 연동 실패가 전체 시스템 장애로 번지면 안 된다.
- GitHub 조회 실패, 문서 추출 실패, AI 생성 실패는 각각 독립된 오류로 식별 가능해야 한다.
- 부분 실패가 발생하더라도 이미 저장된 정상 데이터는 유지되어야 한다.

#### 4.4.3 복구 목표 초안

- 애플리케이션 프로세스 재시작 후 핵심 API 재가동: 15분 이내
- 단순 배포 실패 시 이전 버전 롤백 시작: 30분 이내
- DB 장애 복구 절차는 별도 운영 문서로 구체화한다.

### 4.5 외부 연동 신뢰성

#### 4.5.1 OAuth2 제공자 연동

- OAuth2 제공자 장애 또는 응답 지연 시 로그인 실패 원인을 사용자에게 구분해서 안내해야 한다.
- OAuth2 콜백 처리 실패는 재시도 가능한 형태로 로그를 남겨야 한다.

#### 4.5.2 GitHub 연동

- GitHub API rate limit 초과, 권한 부족, 대상 repository 접근 불가를 구분해서 처리해야 한다.
- repository 조회 실패와 commit 동기화 실패는 별도 상태값으로 관리 가능해야 한다.
- 사용자 본인 commit 식별 실패 시 전체 동기화를 성공으로 처리하지 않고 부분 실패 또는 실패로 식별할 수 있어야 한다.

#### 4.5.3 AI 연동

- AI 응답 실패, timeout, 형식 불일치, 빈 응답은 정상 결과로 저장하면 안 된다.
- 생성 실패 시 재시도 가능 여부를 판단할 수 있도록 실패 사유를 남겨야 한다.
- 모델 fallback 전략을 적용하는 경우, 어떤 모델이 사용되었는지 추적 가능해야 한다.

#### 4.5.4 timeout / retry 기준 초안

- 외부 API 호출은 connect timeout, read timeout을 명시적으로 설정해야 한다.
- 동일 요청의 무한 재시도는 허용하지 않는다.
- idempotent한 조회성 요청은 제한된 횟수의 재시도를 허용할 수 있다.
- 생성성 요청은 중복 저장이 발생하지 않도록 재시도 정책을 별도로 관리해야 한다.

### 4.6 파일 업로드 및 문서 처리 품질

#### 4.6.1 지원 형식

- MVP 기준 우선 지원 형식은 PDF, DOCX, MD이다.
- 지원하지 않는 형식은 업로드 단계에서 차단해야 한다.
- 암호화되었거나 정상 추출이 불가능한 문서는 실패 상태로 처리할 수 있어야 한다.

#### 4.6.2 용량 제한

- 파일 업로드 최대 용량은 수치가 아직 확정되지 않았으므로 별도 결정이 필요하다.
- 초안 제안: 파일 1개당 최대 20MB
- 용량 초과 시 업로드를 차단하고 명확한 오류 메시지를 제공해야 한다.

#### 4.6.3 추출 품질

- 문서 업로드 성공과 텍스트 추출 성공은 별도로 관리해야 한다.
- 추출 실패도 상태값으로 저장되어 이후 재처리 또는 사용자 안내가 가능해야 한다.
- 추출된 텍스트가 비정상적으로 짧거나 비어 있는 경우 품질 저하 경고를 남길 수 있어야 한다.

#### 4.6.4 저장 일관성

- 파일 메타데이터 저장과 실제 스토리지 저장 결과가 불일치하면 안 된다.
- 스토리지 저장 실패 시 DB 메타데이터만 남는 상태를 최소화해야 한다.

### 4.7 데이터 무결성 및 저장 정책

#### 4.7.1 관계 무결성

- DB 설계 문서에 정의된 unique 제약과 참조 관계는 실제 스키마에 반영되어야 한다.
- application, question set, session, answer 간 관계는 잘못 연결되면 안 된다.
- session 종료 전에는 결과 완료 상태로 저장되면 안 된다.

#### 4.7.2 트랜잭션 기준

- application 생성 및 source 연결 저장은 논리적으로 일관된 단위로 처리해야 한다.
- 면접 세션 종료와 결과 저장은 부분 저장으로 인해 completed 상태와 결과 누락이 함께 발생하지 않도록 관리해야 한다.
- 삭제 연산은 cascade/restrict 정책과 실제 서비스 동작이 일치해야 한다.

#### 4.7.3 중복 방지

- 동일 provider 계정 중복 연결, 동일 repository 중복 저장, 동일 commit 중복 저장, 동일 answer-tag 중복 저장은 허용되지 않아야 한다.
- 재시도나 중복 요청으로 같은 application question 또는 answer가 중복 생성되지 않도록 해야 한다.

#### 4.7.4 백업 및 복원 초안

- 운영 환경에서는 최소 일 1회 이상 DB 백업 또는 스냅샷이 가능해야 한다.
- 복원 절차는 운영 가이드 문서에서 검증되어야 한다.

### 4.8 로그, 모니터링, 추적성

#### 4.8.1 로그 기준

- 애플리케이션 로그는 요청 단위 추적이 가능해야 한다.
- requestId 또는 traceId를 모든 주요 요청 흐름에 포함해야 한다.
- 구조화 로그를 기본으로 하며, 로그 레벨을 구분해야 한다.

#### 4.8.2 필수 모니터링 대상

- 로그인 성공/실패 건수
- GitHub 연동 성공/실패 건수
- 문서 업로드 성공/실패 건수
- 텍스트 추출 성공/실패 건수
- 자소서 생성 성공/실패 건수
- 면접 질문 생성 성공/실패 건수
- 면접 세션 시작/종료 건수
- 외부 API timeout 및 error 비율

#### 4.8.3 헬스체크

- 애플리케이션 기본 헬스체크 endpoint가 있어야 한다.
- DB 연결 상태와 핵심 의존성 상태를 확인할 수 있어야 한다.
- readiness/liveness를 구분할 수 있으면 더 바람직하다.

#### 4.8.4 알림 기준 초안

- 특정 시간 동안 오류율이 기준치를 초과하면 운영자가 인지할 수 있어야 한다.
- AI 생성 실패율, GitHub 연동 실패율, 문서 추출 실패율의 급격한 상승은 경고 대상이 되어야 한다.

### 4.9 테스트 및 품질 보증

#### 4.9.1 테스트 범위

- 핵심 도메인 서비스 로직은 단위 테스트가 있어야 한다.
- API 입력 validation과 예외 응답은 테스트 가능해야 한다.
- DB 제약 및 매핑 검증은 통합 테스트로 확인해야 한다.
- 외부 연동 어댑터는 mocking 또는 sandbox 환경을 활용해 테스트 가능해야 한다.

#### 4.9.2 필수 검증 흐름

- OAuth2 로그인 성공/실패 흐름
- GitHub repository 목록 조회 및 commit 동기화 흐름
- 문서 업로드 및 추출 성공/실패 흐름
- 자소서 생성 성공/실패 흐름
- 면접 질문 생성 성공/실패 흐름
- 모의 면접 세션 생성, 답변 저장, 종료, 결과 조회 흐름

#### 4.9.3 CI 기준

- main 또는 develop 머지 전 자동 테스트가 통과해야 한다.
- 정적 분석 또는 포맷 검사가 있다면 CI에서 함께 검증해야 한다.
- 실패한 빌드 결과는 머지 기준을 통과할 수 없어야 한다.

### 4.10 배포 및 운영 기준

#### 4.10.1 환경 분리

- local, dev, prod 환경은 분리된 설정을 사용해야 한다.
- 환경별 DB, secret, 외부 API 설정은 분리 관리해야 한다.

#### 4.10.2 배포 원칙

- 애플리케이션은 재현 가능한 빌드 결과물로 배포되어야 한다.
- Docker 기반 실행 환경을 표준으로 맞추는 것을 권장한다.
- 배포 시 설정 누락, 마이그레이션 실패, health check 실패를 감지할 수 있어야 한다.

#### 4.10.3 DB 변경 관리

- 스키마 변경은 문서 없이 수동 반영하지 않는다.
- DB 변경은 migration 도구 또는 명시적인 DDL 관리 절차를 통해 반영해야 한다.
- 운영 DB 직접 수정은 예외 상황을 제외하고 금지한다.

#### 4.10.4 롤백 가능성

- 신규 배포가 핵심 기능 장애를 유발할 경우 이전 안정 버전으로 되돌릴 수 있어야 한다.
- 최소한 애플리케이션 레벨 롤백 절차는 문서화되어야 한다.

### 4.11 AI 생성 기능 운영 기준

#### 4.11.1 생성 품질 최소 기준

- 자소서 생성과 면접 질문 생성은 선택된 포트폴리오 데이터 또는 application 데이터를 기반으로 해야 한다.
- 입력 데이터가 부족할 때는 생성 실패 또는 품질 저하 안내를 제공해야 한다.
- 빈 응답, 형식이 깨진 응답, 질문 수 불일치 등 비정상 결과는 후처리 단계에서 감지해야 한다.

#### 4.11.2 프롬프트 및 결과 추적성

- 어떤 입력 데이터와 어떤 생성 옵션으로 결과가 만들어졌는지 추적 가능해야 한다.
- 프롬프트 전문 저장 여부는 별도 정책이 필요하지만, 최소한 프롬프트 버전 또는 생성 전략 버전은 관리할 수 있어야 한다.

#### 4.11.3 비용 및 사용량 관리

- 고비용 생성 기능은 사용량 추적이 가능해야 한다.
- 향후 무료/유료 플랜 분리를 고려해 사용자별 생성 횟수 또는 비용 상한을 확장 가능하게 설계해야 한다.

#### 4.11.4 사용자 검토 가능성

- AI 생성 자소서와 면접 결과는 사용자가 검토하고 수정할 수 있어야 한다.
- AI 결과를 사용자 확인 없이 외부 제출 결과로 간주하면 안 된다.

### 4.12 확장성 및 유지보수성

#### 4.12.1 코드 구조

- 인증, GitHub 연동, 문서 처리, 자소서 생성, 면접 세션 도메인은 책임이 분리된 구조로 설계해야 한다.
- 외부 연동과 도메인 로직은 직접 강결합되지 않도록 추상화 계층을 둘 수 있어야 한다.

#### 4.12.2 후속 기능 확장 고려

- PR/Issue 수집, 음성 면접, 결제, 통계 기능이 추가되더라도 핵심 엔티티 구조를 전면 재작성하지 않도록 설계해야 한다.
- enum, 상태값, 태그 체계는 확장 가능한 방식으로 관리하는 것이 바람직하다.
