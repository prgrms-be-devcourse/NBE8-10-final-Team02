## 2. 이 프로젝트에서 AI 문서 주도 개발이 필요한 이유

이 프로젝트는 단순 CRUD 서비스가 아니라 아래 요소가 강하게 얽혀 있다.

- 로그인과 외부 계정 연동이 분리된다.
- GitHub 연동, 문서 업로드, 텍스트 추출, AI 생성, 면접 세션, 결과 저장이 하나의 흐름으로 이어진다.
- 자소서 생성과 면접 질문 생성은 `application` 중심으로 연결된다.
- 화면 상태, API 계약, DB 구조, 오류 정책, AI 출력 스키마가 서로 맞물려야 한다.
- 외부 연동 실패와 내부 도메인 오류를 분리해 다뤄야 한다.
- MVP 범위 밖 기능을 AI가 과하게 추정해 구현할 위험이 있다.

따라서 이 프로젝트에서는 코드보다 문서를 먼저 맞추고, 문서 간 원본 우선순위를 명확히 두며, 변경이 생기면 같은 PR에서 관련 문서를 함께 수정하는 방식이 필요하다.

## 3. 협업 기본 원칙

### 3.1 문서가 코드보다 먼저다

- 기능 범위, API 계약, DB 구조, 오류 정책, AI 출력 형식이 바뀌면 코드를 수정하기 전에 문서를 먼저 맞춘다.
- 문서가 비어 있거나 충돌하면 구현보다 문서 보강을 우선한다.

### 3.2 문서마다 담당 주제를 하나로 유지한다

- 요구사항 명세서는 기능 범위와 수용 기준의 원본이다.
- API 명세서는 HTTP 계약의 원본이다.
- DB 설계 문서는 저장 구조의 원본이다.
- AI 프롬프트 템플릿 설계서는 AI 입출력 계약의 원본이다.
- 공통 개발 규칙은 구현 규칙의 원본이다.
- AGENTS.md는 요약 규칙만 담고, 세부 본문은 다른 문서로 연결한다.

### 3.3 추정 구현을 금지한다

- 문서에 없는 상태값, enum, 에러 코드, 응답 필드, 테이블, 화면을 임의로 추가하지 않는다.
- MVP 제외 범위인 PR/Issue 수집, 음성/영상 면접, 결제, 광고, 커뮤니티, 대규모 통계는 문서 메모 없이 구현하지 않는다.

### 3.4 같은 변경은 같은 PR에서 닫는다

아래가 바뀌면 관련 문서를 같은 PR에서 함께 수정한다.

- 기능 범위
- 요청/응답 구조
- 인증 방식
- DB 컬럼/제약/인덱스
- 화면 상태/라우트
- 예외 코드와 메시지 구조
- AI 출력 스키마
- 테스트 기준

### 3.5 문서는 설명, 강제는 자동화가 맡는다

- 문서만으로 운영하지 않는다.
- CODEOWNERS, PR 템플릿, CI, 린트, 테스트, 브랜치 보호 규칙을 함께 둔다.
- `openapi.yaml`, migration 파일, 워크플로 파일처럼 기계 판독 가능한 파일이 생기면 해당 파일을 더 높은 원본으로 본다.

## 5. 문서 충돌 시 최종 판단 규칙

문서 전체 우선순위는 아래 원칙을 따른다.

### 5.1 기능 범위 충돌

- 요구사항 명세서를 우선한다.
- 프로젝트 기획서나 WBS에 적혀 있어도 요구사항 명세서에 없는 기능은 구현하지 않는다.

### 5.2 HTTP 계약 충돌

- `openapi.yaml`이 생기면 `openapi.yaml`을 최우선으로 본다.
- 그 전까지는 API 명세서를 우선한다.
- 화면 정의서의 필드명이나 와이어프레임 표현이 API 명세서와 다르면 API 명세서를 기준으로 화면 문서를 수정한다.

### 5.3 DB 구조 충돌

- migration 파일이 생기면 migration 파일을 최우선으로 본다.
- migration 파일이 없을 때는 DB 설계 정리를 ERD보다 우선한다.
- ERD는 개념 관계 설명용으로 사용하고, 실제 컬럼/제약/인덱스는 DB 설계 정리를 따른다.

### 5.4 화면 동작 충돌

- 화면 상태, 라우트, 진입 조건은 화면 정의서를 우선한다.
- 와이어프레임은 배치와 흐름 설명용으로 사용한다.
- 와이어프레임과 화면 정의서가 다르면 와이어프레임을 수정한다.

### 5.5 AI 출력 충돌

- AI 템플릿의 입력 변수, 출력 JSON 구조, 재시도 규칙은 AI 프롬프트 템플릿 설계서를 우선한다.
- API 응답 예시나 DB 저장 구조가 다르면 AI 템플릿 문서와 API/DB 문서를 함께 수정한다.

### 5.6 오류 정책 충돌

- 오류 분류 원칙과 사용자 메시지/로그 메시지 분리는 예외 및 오류 처리 정책을 우선한다.
- endpoint별 실제 응답 예시는 API 명세서를 우선한다.
- 둘이 충돌하면 오류 정책 문서의 분류 기준을 유지하면서 API 명세 예시를 수정한다.

### 5.7 구현 규칙 충돌

- 공통 개발 규칙을 우선한다.
- AGENTS.md는 공통 개발 규칙의 요약본으로만 둔다.
- AGENTS.md가 공통 개발 규칙과 다르면 AGENTS.md를 수정한다.

### 5.8 일정 문서의 한계

- WBS는 일정과 산출물 계획용 문서이다.
- WBS는 기능 상세, API 계약, DB 구조의 원본이 아니다.
- 구현 판단의 근거로 WBS만 단독 사용하지 않는다.

## 10. AI 작업 중단 조건

아래 중 하나라도 해당하면 AI는 코드 생성을 계속하지 않고 먼저 문서 보강 또는 확인 요청으로 전환한다.

### 10.1 상위 문서가 비어 있는 경우

- 요구사항 명세서에 기능이 없는데 API만 존재하는 경우
- 화면 정의서에 상태 정의가 없는데 화면 API가 먼저 필요한 경우
- AI 템플릿 출력 스키마가 없는데 저장 구조를 만들려는 경우

### 10.2 미결정 항목을 건드리는 경우

아래 항목은 현재 열어둔 것으로 취급한다.

- 업로드 허용 형식 최종 범위
- 업로드 용량 제한 수치
- GitHub private repository 지원 시점
- 자소서 생성 분량 제한 정책
- AI 생성 재시도 정책의 최종 수치
- 면접 점수 산정 방식의 상세 기준
- 약점 태그 분류 체계의 상세 목록
- `applications.status` 세분화 여부
- 태그 마스터 운영 방식

### 10.3 새 enum, 상태값, 오류코드가 필요한 경우

- 기존 문서에 없는 enum, 상태값, 오류코드는 구현 전에 문서에 먼저 추가한다.
- 특히 `application_status`, `interview_session_status`, `extract_status`, 오류 코드 문자열은 임의 확장을 금지한다.

### 10.4 문서 간 충돌이 있는 경우

예시
- 화면 정의서 라우트와 API 명세 path가 상충
- ERD 관계와 DB unique 제약이 상충
- 요구사항 예외 케이스와 오류 정책의 상태코드 해석이 상충
- AI 템플릿 출력 필드와 DB 저장 필드가 상충

### 10.5 MVP 제외 범위를 구현하려는 경우

- PR/Issue 수집
- 음성/영상 면접
- 결제/광고/토큰 시스템
- 커뮤니티/게임화/대규모 통계
- 비동기 잡 로그, 비용 로그, 운영 감사 로그 상세

## 11. 문서 상태와 책임자 운영 규칙

### 11.1 문서 상태

각 문서는 아래 상태 중 하나를 가진다.

- `draft`: 초안, 구현 기준으로 쓰기 전
- `reviewed`: 팀 검토 완료, 구현 기준 사용 가능
- `frozen`: 현재 스프린트 동안 변경 금지, 긴급 변경만 허용
- `deprecated`: 더 이상 원본으로 사용하지 않음

### 11.2 문서 메타데이터 권장 필드

각 문서 상단에 아래 항목을 추가하는 것을 권장한다.

- owner
- reviewer
- status
- last_updated
- linked_issue_or_pr
- applies_to

### 11.3 역할 분담 기준 추천

| 역할 | 우선 소유 문서 |
|---|---|
| 플랫폼/공통 기반 + 인프라/배포/관측성 | 공통 개발 규칙, 비기능 요구사항, 배포/운영 문서, AGENTS.md |
| 포트폴리오 수집/외부 연동 + 품질/테스트 | GitHub 관련 문서, 문서 업로드/추출 정책, 테스트 문서 |
| AI 기능/생성 파이프라인 | AI 프롬프트 템플릿 설계서, AI 기능 관련 API/오류 정책 |
| 면접 세션/서비스 흐름 + 대시보드/히스토리 | 화면 정의서, 와이어프레임, 면접 흐름 관련 요구사항/API |

## 12. PR 체크리스트에 반드시 넣을 항목

아래 항목은 PR 템플릿에 포함하는 것을 권장한다.

- 이 변경의 상위 원본 문서는 무엇인가
- 요구사항 변경이 있는가
- API 계약 변경이 있는가
- DB 구조 또는 제약 변경이 있는가
- 오류 코드/상태코드 변경이 있는가
- 화면 상태/라우트 영향이 있는가
- AI 템플릿 또는 출력 스키마 변경이 있는가
- 관련 테스트를 추가했는가
- 관련 문서를 같은 PR에서 함께 수정했는가

## 13. 현재 기준으로 추가하면 좋은 문서와 파일

현재 문서 세트는 상당히 좋지만, 실제 AI 협업을 더 안정화하려면 아래가 추가되면 좋다.

### 13.1 꼭 추가 권장

- `README.md`
- `AGENTS.md`
- `CONTRIBUTING.md`
- `openapi.yaml`
- `CODEOWNERS`
- `.github/pull_request_template.md`
- `.github/workflows/*.yml`

### 13.2 강하게 추천

- `docs/testing.md`
- `docs/security.md`
- `docs/observability.md`
- `docs/db-migration.md`
- `docs/adr/*`
- `docs/deployment-runbook.md`

### 13.3 현재 문서에서 분리 고려 가능

- 예외 및 오류 처리 정책에서 `error-codes.md` 분리
- 비기능 요구사항에서 `security.md`, `observability.md`, `testing.md` 분리
- API 명세서 md에서 `openapi.yaml` 분리

## 14. 지금 바로 실행할 권장 순서

1. 루트 `AGENTS.md`를 만든다.
2. `openapi.yaml` 초안을 만든다.
3. PR 템플릿과 CODEOWNERS를 추가한다.
4. 본 문서의 추적 매트릭스를 기준으로 FR-화면-API-DB 연결 누락을 점검한다.
5. 현재 `draft` 문서 중 이번 스프린트 구현 범위에 필요한 문서를 `reviewed`로 승격한다.
6. 미결정 항목 중 구현에 직접 영향 주는 항목만 먼저 좁힌다.

## 15. 루트 AGENTS.md에 넣을 핵심 요약 초안

아래 내용은 루트 `AGENTS.md`에 요약본으로 넣기 좋다.

```md
# AGENTS.md

## 먼저 읽을 문서
1. 요구사항 명세서
2. API 명세서 또는 openapi.yaml
3. DB 설계 정리
4. 공통 개발 규칙
5. 해당 디렉터리 하위 AGENTS.md

## 핵심 규칙
- 없는 요구사항을 추정해 구현하지 않는다.
- 문서가 바뀌면 코드를 고치기 전에 문서를 먼저 맞춘다.
- Entity를 응답으로 직접 노출하지 않는다.
- 외부 API 호출을 긴 트랜잭션 안에 넣지 않는다.
- API/DB/오류코드 변경은 같은 PR에서 문서를 함께 수정한다.
- openapi.yaml이 생기면 API md보다 openapi를 우선한다.
- migration 파일이 생기면 DB 문서보다 migration을 우선한다.

## 작업 완료 전 확인
- 관련 테스트를 추가했는가
- 문서를 함께 수정했는가
- 새 enum/상태값/오류코드를 임의로 만들지 않았는가
- 로그에 민감정보가 남지 않는가
```

## 16. 문서 작성 스타일 원칙

- 각 문서는 하나의 주제만 담당한다.
- 적용 범위를 문서 상단에 명시한다.
- MUST와 SHOULD를 구분한다.
- 애매한 표현을 피한다.
- 실제 경로, 실제 필드명, 실제 enum 이름을 쓴다.
- 예시 응답과 금지 예시를 함께 둔다.
- 같은 내용을 여러 문서에 중복 복사하지 않고, 원본 문서를 정한 뒤 링크한다.

## 17. 결론

이 프로젝트는 이미 좋은 설계 문서를 많이 가지고 있다. 지금 필요한 것은 문서를 더 많이 만드는 것보다, 현재 있는 문서들이 언제 원본이 되고, 어떤 변경이 생기면 어떤 문서를 같이 고쳐야 하는지, AI가 어디까지 구현하고 어디서 멈춰야 하는지를 명확히 운영 규칙으로 묶는 일이다.

이 문서는 그 운영 규칙의 시작점으로 사용한다.
