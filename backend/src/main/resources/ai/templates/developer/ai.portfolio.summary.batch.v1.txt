목표:
- 복수의 repository 데이터에서 각 repo별 재사용 가능한 evidence pack을 추출한다.
- 이 출력은 자소서·면접 질문 생성 시 job 컨텍스트와 결합된다.
- job에 맞게 해석하는 것은 다음 단계의 역할이므로, 여기서는 사실에 기반한 범용 evidence를 충실히 담는다.
- 각 repository를 독립적으로 분석하여 독립적인 결과를 만든다. repo 간 통합 분석은 절대 하지 않는다.

---

입력 포맷:

XML <batch_data> 태그 안에 복수의 <repository> 태그가 포함된다.
<repository id="{repoId}" name="{repoName}" type="own|team|external_org|opensource">
  <overview>README 텍스트 (있는 경우)</overview>
  <code_structure>기여 파일 목록, PageRank 포함</code_structure>
  <diffs>
    <diff phase="early">초기 커밋 (커밋 sha, subject, body, diff 포함)</diff>
    <diff phase="mid">중기 커밋</diff>
    <diff phase="late">후기 커밋</diff>
  </diffs>
</repository>

각 섹션 활용 지침:
- <overview>: 프로젝트 전체 맥락 파악. stack·role 추론에 활용.
- <code_structure>: [authored] 표시된 파일이 본인 기여 파일. import 분석 기반 stack 추출, role 도메인 추론.
- <diff phase="early">: 초기 설계·아키텍처 커밋. projectKey/projectName 힌트 포함.
- <diff phase="mid">: 핵심 구현 커밋. evidenceBullets 소재.
- <diff phase="late">: 최적화·트러블슈팅 커밋. challenges 소재.

커밋 분류 기준 (각 repo에 독립 적용):
- evidenceBullets 소재: 기능 구현·성능 개선 커밋 (feat/perf 계열)
- challenges 소재: 결함 수정·장애 해결 커밋 (fix/resolve 계열, 에러 키워드 포함)
- techDecisions 소재: body 또는 diff에 // WHY: / // NOTE: 근거가 있는 커밋
                      (근거 없으면 반드시 빈 배열 — 생성 금지)

---

출력 포맷:

반드시 JSON 배열로만 응답한다. 원소 수는 입력 <repository> 태그 수와 정확히 같아야 한다.
배열의 순서는 입력 <repository> 순서와 같아야 한다.
입력 <repository>가 1개뿐이어도 반드시 배열로 감싸야 한다: [{...}]
절대로 배열 없이 객체 단독({...})으로 응답하지 말 것.

[
  {
    "repoId": "<repository id 속성값 그대로>",
    "project": { ...아래 schema... }
  }
]

금지 사항:
- 여러 repo를 합친 통합 분석(글로벌 리스크, 전체 강점 등)을 생성하지 말 것.
- 입력에 없는 repo를 새로 만들어내지 말 것.
- repo 경계를 넘어 다른 repo의 정보를 혼용하지 말 것 (Context Bleeding 금지).
- 배열 외부에 추가 설명 텍스트, 마크다운, 서론을 출력하지 말 것.

---

각 project 출력 필드 작성 지침:

stack
  해당 repository의 <code_structure>에서 확인된 기술/프레임워크.
  "어떤 기술을 사용했냐" — 기술명·프레임워크명만.
  예) Java 17, Spring Boot 3, JPA, Redis, PostgreSQL

signals
  해당 repository의 <diffs>에서 드러나는 구현·해결 행위 키워드.
  "무엇을 구현/해결했냐" — 기술명이 아닌 행위 중심으로 표현.
  예) OAuth2 소셜 로그인, 분산 락, GitHub API 연동
  stack과 중복 금지.

role
  해당 repository의 <code_structure>에서 [authored] 파일 패턴 전체를 보고 추론.
  단순 커밋 수 기준이 아닌, 어떤 도메인/레이어를 주로 담당했는지 판단.

evidenceBullets
  해당 repository의 feat/perf 커밋 diff에서 추출. 1항목 = 1문장.
  challengeRef: 파생된 challenge.id, 없으면 null.

challenges
  해당 repository의 fix/resolve 커밋에서 추출.
  what/how/learning 형식. evidenceBullets에 근거 없는 challenge 금지.

techDecisions
  해당 repository의 커밋 body 또는 // WHY: / // NOTE: 근거가 있는 커밋에서만 추출.
  근거 없으면 반드시 빈 배열. hallucination 금지.

strengths / risks
  해당 repository 입력만을 기반으로 판단. 다른 repo 정보 혼용 금지.
  구체적 근거 없이 일반적인 표현 금지.

---

출력 규칙:
- 반드시 JSON 배열만 출력한다. 설명 텍스트, 마크다운 코드 블록, 서론 없음.
- 각 project의 signals는 최대 6개, evidenceBullets는 최대 5개, challenges는 최대 3개, techDecisions는 최대 3개.
- 근거가 약하면 qualityFlags에 "low_context" 또는 "weak_evidence"를 넣는다.
- 배열 원소 수는 반드시 입력 <repository> 수와 동일해야 한다.

각 project JSON 스키마 (필드명을 정확히 사용할 것):
{
  "projectKey": "string — 영문 소문자+하이픈",
  "projectName": "string — 사람이 읽기 좋은 이름",
  "summary": "string — 2~3문장 요약",
  "role": "string — 예: 백엔드 개발 / 인증 도메인 담당",
  "stack": ["string"],
  "signals": ["string"],
  "evidenceBullets": [
    { "fact": "string", "challengeRef": "string | null" }
  ],
  "challenges": [
    { "id": "string", "what": "string", "how": "string", "learning": "string" }
  ],
  "techDecisions": [
    { "decision": "string", "reason": "string", "tradeOff": "string | null" }
  ],
  "strengths": ["string"],
  "risks": ["string"],
  "sourceRefs": ["string"],
  "qualityFlags": ["low_context | weak_evidence"]
}
