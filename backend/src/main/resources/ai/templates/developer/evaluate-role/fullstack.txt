[직무별 평가 기준: 풀스택 개발자]

이 답변자의 직무는 풀스택 개발자이다. 아래 기준을 평가에 추가 적용한다.

domainDepth 축 해석:
- 프론트엔드-백엔드 간 연동 설계, API 계약, 데이터 흐름의 전체적 이해에 대한 깊이를 중점 평가한다.
- experience 질문: 프론트와 백엔드를 아우르는 E2E 문제 해결 과정과 양쪽 관점에서의 의사결정을 기대한다.
- technical_stack 질문: 프론트엔드와 백엔드 기술 조합의 일관성과 전체 아키텍처 관점의 trade-off를 기대한다.
- technical_cs 질문: 네트워크 통신, 직렬화/역직렬화, 인증/인가 흐름 등 프론트-백 경계에서의 이해를 기대한다.

직무 특화 참고 예시 (이 예시 자체를 출력에 포함하지 말 것):

[풀스택 - 좋은 답변 평가 - technical_stack]
질문(technical_stack): "프로젝트의 프론트엔드와 백엔드 기술 스택을 어떻게 결정했나요?"
답변: "백엔드는 Spring Boot로 REST API를 구성하고, 프론트엔드는 Next.js의 SSR을 활용했습니다. API 계약을 OpenAPI 스펙으로 먼저 정의한 뒤 코드 생성기로 타입 안전성을 확보했고, BFF 패턴으로 프론트 전용 API 레이어를 두어 백엔드 도메인 모델이 프론트에 노출되지 않도록 분리했습니다."
→ score: 82
→ rationale: "①OpenAPI 기반 계약 우선 접근과 BFF 패턴으로 프론트-백 경계를 명확히 설계한 아키텍처 판단이 체계적이다. ②SSR 도입에 따른 서버 부하 증가와 캐싱 전략, BFF 레이어의 유지보수 비용에 대한 고려가 언급되지 않았다. ③SSR 페이지의 캐시 정책(ISR, CDN 캐싱)과 BFF 레이어가 장기적으로 N+1 API 호출 문제를 어떻게 방지하는지 설명하라."
→ tags: ["기술 깊이 부족"]

[풀스택 - 부족한 답변 평가 - technical_stack]

답변: "백엔드는 Spring Boot, 프론트는 React를 사용했습니다. 둘 다 인기가 많고 자료가 풍부해서 선택했습니다."

→ rationale: "①주류 기술 스택을 선택했다는 점은 인지했다. ②프로젝트의 요구사항이나 제약 조건에 기반한 기술적 판단이 전혀 없으며, 프론트-백 연동 방식에 대한 설계 고려가 부재하다. ③API 통신 방식(REST vs GraphQL), 렌더링 전략(CSR vs SSR), 타입 공유 방식 등 연동 관점의 의사결정 근거를 포함하라."
