[직무별 평가 기준: 백엔드 개발자]

이 답변자의 직무는 백엔드 개발자이다. 아래 기준을 평가에 추가 적용한다.

domainDepth 축 해석:
- 서버 아키텍처, DB 설계/최적화, API 설계, 동시성/스레드 관리, 캐싱 전략에 대한 깊이를 중점 평가한다.
- experience 질문: 서버 사이드 성능 지표(TPS, 응답시간, DB 쿼리 성능)의 구체적 수치와 개선 과정을 기대한다.
- technical_stack 질문: 프레임워크/DB/메시징 시스템 선택의 trade-off와 대안 비교를 기대한다.
- technical_cs 질문: 자료구조, 알고리즘의 시간/공간 복잡도 분석과 서버 환경에서의 적용 사례를 기대한다.

직무 특화 참고 예시 (이 예시 자체를 출력에 포함하지 말 것):

[백엔드 - 좋은 답변 평가 - technical_stack]
질문(technical_stack): "왜 PostgreSQL을 선택했나요?"
답변: "초기에 MySQL과 비교 검토했습니다. JSONB 컬럼으로 유연한 스키마 확장이 가능하고, 윈도우 함수와 CTE 지원이 복잡한 집계 쿼리에 유리했습니다. 다만 커넥션 비용이 MySQL보다 높아 PgBouncer로 커넥션 풀링을 적용했습니다."
→ score: 84
→ rationale: "①JSONB와 윈도우 함수라는 구체적 기능 기준으로 MySQL과의 비교를 수행하고, 단점까지 인지하여 PgBouncer로 보완한 판단이 체계적이다. ②선택 후 운영 과정에서 발생한 구체적 이슈(예: 대용량 JSONB 인덱싱, 버큠 튜닝)에 대한 경험이 부재하다. ③실제 운영에서 JSONB 인덱스 전략(GIN vs GiST)이나 autovacuum 설정 등 PostgreSQL 특유의 운영 노하우를 보완하라."
→ tags: ["기술 깊이 부족"]

[백엔드 - 부족한 답변 평가 - technical_stack]

→ score: 30
→ rationale: "①기존 기술 스택을 유지했다는 맥락은 제시했다. ②본인의 기술적 판단이 전혀 없으며, 왜 PostgreSQL이 해당 프로젝트에 적합한지에 대한 분석이 부재하다. ③MySQL, MongoDB 등 대안과의 비교 관점(ACID 보장, 확장성, 쿼리 복잡도)을 포함하여 기술 선택의 근거를 재구성하라."
→ tags: ["선택 근거 부족", "구체성 부족"]
