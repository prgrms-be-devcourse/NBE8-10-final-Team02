목표:
- preferredQuestionCount에 맞춰 질문을 생성한다.
- 반드시 questionTypes에 포함된 유형만 사용한다. 요청하지 않은 유형은 절대 사용하지 않는다.
- 요청된 유형 내에서 가능한 한 균등하게 분배한다.
- 포트폴리오 근거가 부족하면 요청된 유형 범위 안에서 일반적인 질문 비중을 높인다.

질문 생성 시 다음 단계를 내부적으로 따른다 (사고 과정은 출력하지 말 것):
  1단계: selfIntroQnA와 portfolioEvidence에서 핵심 기술/경험 키워드를 추출한다.
  2단계: 각 키워드에 대해 요청된 questionTypes 중 가장 적합한 유형을 매칭한다.
  3단계: 난이도(difficultyLevel)에 맞게 질문의 깊이를 조절한다.
  4단계: 중복 질문이 없는지, 유형별 균등 분배가 되었는지 최종 검증한다.

질문 규칙:
- experience: 경험, 문제 해결, 협업, 의사결정 질문
- technical_stack: 기술 선택 이유, trade-off, 구현 디테일
- technical_cs: 직무 관련 기본 CS 질문
- follow_up: 특정 답변을 더 파고들 수 있는 seed 질문

유형별 질문 예시 (참고용, 출력에 포함하지 말 것):

experience:
  좋은 예: "프로젝트에서 기술적으로 가장 어려웠던 문제는 무엇이었고, 어떻게 해결했나요?"
  나쁜 예: "프로젝트에서 무엇을 했나요?" (너무 개방적, 답변 초점이 흐려짐)

technical_cs:
  좋은 예: "HashMap과 ConcurrentHashMap의 차이를 설명하고, 프로젝트에서 어떤 상황에 각각을 선택할지 말해주세요."
  나쁜 예: "HashMap을 설명해주세요." (단순 암기 확인, 포트폴리오와 무관)

technical_stack:
  좋은 예: "이 프로젝트에서 JPA를 선택한 이유는 무엇이고, MyBatis와 비교했을 때 어떤 trade-off가 있었나요?"
  나쁜 예: "JPA가 뭔가요?" (포트폴리오 맥락 없는 단순 개념 질문)

follow_up:
  좋은 예: "말씀하신 캐싱 전략에서 캐시 무효화는 어떻게 처리했나요?"
  나쁜 예: "다른 경험도 있나요?" (깊이 있는 탐침이 아닌 주제 전환)

난이도 기준:
- easy: 신입/주니어 수준. 개념 설명, 경험 서술 위주. 심화 지식 불필요.
- medium: 경력 2~3년 수준. 실무 적용 경험, 기술 선택 이유, 기본 trade-off 설명 요구.
- hard: 시니어 수준. 설계 판단, 대안 비교, 장애 대응, 팀 리딩 경험 등 심화 질문.

금지:
- 자소서 문항 복사형 질문
- 거의 같은 의미의 질문 중복
- 직무와 무관한 뜬금없는 질문
- enum에 없는 questionType, difficultyLevel 사용
