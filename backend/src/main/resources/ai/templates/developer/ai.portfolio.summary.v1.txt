목표:
- repository 1개의 기여 데이터에서 재사용 가능한 evidence pack을 추출한다.
- 이 출력은 자소서·면접 질문 생성 시 job 컨텍스트와 결합된다. job에 맞게 해석하는 것은 다음 단계의 역할이므로 여기서는 사실에 기반한 범용 evidence를 최대한 충실히 담는다.

---

입력 섹션 (Runtime — user message에 named section으로 전달):

## Repository
  repo 기본 정보: 이름, URL, 크기, author email

## Project Overview (README)
  README 전문 (있는 경우). 프로젝트 전체 맥락 파악용.

## Code Structure (authored classes highlighted)
  기여 파일 목록 (authored_by_me=true)과 각 파일의 PageRank.
  상위 클래스/인터페이스 구조 요약. import 분석 기반 기술 스택 후보.

## Contribution Commits
  docs/chore/style/ci/test/build 등 노이즈 커밋이 사전 제거된 본인 기여 커밋 목록.
  각 커밋 구성: sha, subject(첫 줄), body(커밋 메시지 본문), diff.

  아래 분류 기준으로 커밋을 직접 분류하여 각 출력 필드의 소재로 활용한다:
  - evidenceBullets 소재: 기능 구현·성능 개선 커밋 (feat/perf 계열)
  - challenges 소재:      결함 수정·장애 해결 커밋 (fix/resolve 계열, 또는 에러 키워드 포함)
  - techDecisions 소재:   body 또는 diff에 // WHY: / // NOTE: 근거가 있는 커밋
                          (근거 없으면 빈 배열 — 생성 금지)

---

출력 필드 작성 지침:

stack
  ## Code Structure의 import 분석과 PageRank 상위 파일에서 확인된 기술/프레임워크.
  "어떤 기술을 사용했냐" — 기술명·프레임워크명만.
  예) Java 17, Spring Boot 3, JPA, Redis, PostgreSQL

signals
  ## Contribution Commits에서 드러나는 구현·해결 행위 키워드.
  "무엇을 구현/해결했냐" — 기술명이 아닌 행위 중심으로 표현.
  예) OAuth2 소셜 로그인, 분산 락, GitHub API 연동, 페이지네이션 최적화
  stack과 중복 금지.

role
  ## Code Structure의 기여 파일 패턴 전체를 보고 추론.
  단순 커밋 수 기준이 아닌, 어떤 도메인/레이어를 주로 담당했는지 판단.
  예) "백엔드 개발 / 인증·포트폴리오 도메인 담당"

evidenceBullets
  ## Contribution Commits 중 기능 구현·성능 개선 커밋의 diff에서 추출. 1항목 = 1문장.
  challengeRef: 이 항목에서 파생된 challenge의 id. 없으면 null.

challenges
  ## Contribution Commits 중 결함 수정·장애 해결 커밋에서 추출.
  what: 어떤 문제였나 / how: 어떻게 해결했나 / learning: 무엇을 배웠나.
  evidenceBullets에 근거 없는 challenge는 작성 금지.

techDecisions
  ## Contribution Commits 중 body 또는 diff에 // WHY: / // NOTE: 근거가 있는 커밋에서만 추출.
  커밋 body나 주석에 근거가 없으면 빈 배열. hallucination 금지 — 추측으로 채우지 말 것.

strengths / risks
  전체 입력을 종합해 판단. 구체적 근거 없이 일반적인 표현 금지.

---

출력 규칙:
- 반드시 아래 JSON 스키마를 정확히 따른다. 필드명 오타나 snake_case 변형 금지.
- signals는 최대 6개, evidenceBullets는 최대 5개, challenges는 최대 3개, techDecisions는 최대 3개로 제한한다.
- 모든 signals 키워드는 반드시 evidenceBullets 중 최소 하나의 fact에서 설명되어야 한다.
- 모든 challenges는 evidenceBullets 중 비중 있는 항목에서 파생되어야 한다.
  challengeRef가 있으면 해당 challenge.id가 반드시 challenges 배열에 존재해야 한다.
- 근거가 약하면 qualityFlags에 "low_context" 또는 "weak_evidence"를 넣는다.
- 설명 텍스트 없이 JSON만 출력한다.

출력 JSON 스키마 (필드명을 정확히 사용할 것):
{
  "project": {
    "projectKey": "string — 영문 소문자+하이픈, 예: linebot-backend",
    "projectName": "string — 사람이 읽기 좋은 프로젝트 이름",
    "summary": "string — 2~3문장 요약",
    "role": "string — 본인이 맡은 역할, 예: 백엔드 개발 / 인증 도메인 담당",
    "stack": ["string — 기술/프레임워크명, 예: Java 17, Spring Boot 3"],
    "signals": ["string — 구현/해결 행위, 예: OAuth2 소셜 로그인, 분산 락"],
    "evidenceBullets": [
      {
        "fact": "string — 구체적 기여 항목 1문장",
        "challengeRef": "string | null — 파생된 challenge.id, 없으면 null"
      }
    ],
    "challenges": [
      {
        "id": "string — 짧은 식별자, 예: c1",
        "what": "string — 어떤 문제/난관이었나",
        "how": "string — 어떻게 해결했나",
        "learning": "string — 무엇을 배웠나"
      }
    ],
    "techDecisions": [
      {
        "decision": "string — 어떤 기술/방식을 선택했나",
        "reason": "string — 왜 선택했나 (대안 대비 이유)",
        "tradeOff": "string | null — 트레이드오프, 없으면 null"
      }
    ],
    "strengths": ["string — 이 repo에서 드러나는 강점"],
    "risks": ["string — 약점 또는 개선 필요 항목"],
    "sourceRefs": ["파일경로 또는 커밋SHA"],
    "qualityFlags": ["low_context | weak_evidence"]
  }
}
