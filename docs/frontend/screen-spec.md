# AI 기술 면접 연습 플랫폼 화면 정의서

## 1. 문서 목적

이 문서는 최신 `docs/requirements.md`, `docs/project/open-items.md`, `openapi.yaml`을 기준으로
MVP 화면 구조, 라우트, 상태, 검증 규칙, 화면 간 전환 기준을 정의한다.

이 문서의 목표는 아래 두 가지다.

- 프론트와 백엔드가 같은 사용자 흐름과 상태 전이를 기준으로 구현하게 한다.
- 요구사항에 추가된 제약이 실제 화면에 어떻게 보이는지 명확히 한다.

## 2. 기준 문서

- `docs/requirements.md`
- `docs/project/open-items.md`
- `openapi.yaml`
- `docs/error-policy.md`
- `docs/domain/application.md`
- `docs/domain/interview.md`

## 3. 이번 버전에서 화면에 반드시 드러나야 하는 정책

### 3.1 포트폴리오 관련

- GitHub URL 입력만으로는 public repository만 조회한다.
- private repository는 GitHub OAuth 추가 동의와 적절한 scope가 있을 때만 조회한다.
- PR, Issue 수집은 이번 MVP 화면 범위에 넣지 않는다.
- 업로드 문서는 PDF, DOCX, MD만 허용한다.
- 업로드 문서는 파일당 10MB, 총 5개까지 허용한다.
- 스캔 PDF는 추출 실패 가능성을 사용자에게 안내한다.
- 동일 문서 재업로드 시 덮어쓰기 확인이 필요하다.

### 3.2 지원 준비와 자소서 관련

- 자소서 생성 단위는 `application`이며 화면 표기는 `지원 준비`를 사용한다.
- 회사명, 직무명, 지원 유형 입력 후 문항을 관리한다.
- 문항은 한 번에 최대 10개까지 생성 대상으로 선택한다.
- 분량 옵션은 `short`, `medium`, `long` 3단계다.
- 직무별 기본 템플릿을 제공하고, 회사별 특수 문항은 직접 입력한다.

### 3.3 면접 관련

- 면접 질문은 1회 최대 20개까지 생성한다.
- 실제 세션은 3개 이상 20개 이하 질문으로 구성한다.
- 일반 답변은 50자 이상 1000자 이하를 기준으로 검증한다.
- `건너뛰기`는 별도 액션으로 제공하고, 최소 글자수 규칙 예외로 처리한다.
- 사용자 1명당 동시에 진행 가능한 활성 세션은 1개다.
- 활성 세션은 `in_progress`와 `paused`를 포함한다.
- 브라우저 이탈 또는 새로고침 후 30분 이내에는 세션 복원이 가능하다.
- 30분 이상 미입력 시 세션을 자동 일시정지 상태로 전환한다.

### 3.4 인증 관련

- 로그인 제공자는 GitHub, Google, Kakao 3개를 제공한다.
- 동일 이메일의 다른 소셜 로그인은 자동 병합하지 않는다.
- 기존 계정 안내 또는 별도 계정 연결 절차로 유도한다.

## 4. 전역 UX 원칙

### 4.1 상태 표현

모든 주요 화면은 아래 상태를 공통으로 가져야 한다.

- 초기 로딩
- 빈 상태
- 부분 성공
- 전체 실패
- 재시도 가능
- 접근 불가 또는 선행 조건 미충족

### 4.2 파괴적 액션

아래 액션은 확인 모달이 필요하다.

- 저장소 선택 해제
- 업로드 문서 삭제
- 기존 문서 덮어쓰기
- 지원 준비 삭제
- 면접 세션 종료
- 계정 탈퇴

### 4.3 비동기 작업 노출

아래 작업은 진행 상태를 사용자에게 보여줘야 한다.

- GitHub repository 조회
- commit 수집
- 문서 업로드 및 텍스트 추출
- 자소서 생성
- 면접 질문 생성
- 피드백 리포트 생성

### 4.4 공통 오류 문구 기준

- 이유를 모를 때는 `다시 시도해주세요`만 보여주지 말고 재시도 버튼과 함께 현재 상태를 보여준다.
- 선행 데이터 부족이면 부족한 데이터를 구체적으로 안내한다.
- 권한 부족이면 다시 로그인 또는 추가 동의가 필요한 이유를 함께 안내한다.

## SCR-01 로그인 화면

### 목적
OAuth2 기반 소셜 로그인을 시작한다.

### 권장 라우트
- `/login`

### 진입 조건
- 비로그인 사용자
- 리프레시 실패 등으로 재로그인이 필요한 사용자

### 주요 구성요소
- 서비스 소개 문구
- GitHub 로그인 버튼
- Google 로그인 버튼
- Kakao 로그인 버튼
- 이용약관/개인정보 링크
- 로그인 오류 알림 영역

### 주요 액션
- 제공자 선택 후 OAuth 인증 시작

### 상태
- 기본: 3개 버튼 활성화
- 처리 중: 선택한 버튼 로딩, 나머지 비활성화
- 실패: 인라인 에러와 재시도 버튼
- 중복 계정 안내: 동일 이메일의 다른 소셜 로그인 시도 시 안내 모달 또는 전용 안내 배너

### 검증/예외
- 제공자 취소 시 현재 화면 유지
- 제공자 실패 시 에러 메시지 노출
- 동일 이메일 다른 제공자 로그인은 자동 병합하지 않음

### 완료 후 이동
- 신규 사용자: SCR-02
- 기존 사용자: 포트폴리오 존재 여부에 따라 SCR-03 또는 SCR-08

### 목적
서비스 전체 흐름을 보여주고 포트폴리오 등록으로 진입시킨다.

### 권장 라우트
- `/onboarding`

### 주요 구성요소
- 4단계 흐름 카드: 포트폴리오 등록 → 자소서 생성 → 질문 생성 → 모의 면접
- `GitHub 연결하기` 버튼
- `GitHub URL로 시작하기` 버튼
- `문서 업로드하기` 버튼
- `나중에 하기` 버튼

### 주요 액션
- 포트폴리오 등록 시작
- 건너뛰기

### 상태
- 기본: 단계 카드와 CTA 노출
- 이미 포트폴리오 있음: 건너뛰기 없이 SCR-03 또는 SCR-08로 이동 유도

### 완료 후 이동

## SCR-03 포트폴리오 홈

### 목적
현재 포트폴리오 준비 상태를 요약해서 보여주고 다음 액션으로 연결한다.

### 권장 라우트
- `/portfolio`

### 주요 구성요소
- 프로필 요약 카드
- GitHub 연결 상태 카드
- 선택된 repository 수
- 최근 수집 commit 수
- 업로드 문서 수/추출 성공 여부
- `GitHub 연결 관리`, `문서 업로드`, `지원 준비 시작` CTA
- 최근 실패 작업 알림 영역

### 주요 액션
- GitHub 연결 관리
- 문서 업로드
- 지원 준비 생성
- 기존 포트폴리오 수정

### 상태
- 빈 상태: 아직 repository와 문서가 없으면 온보딩형 안내
- 부분 완료: GitHub만 있거나 문서만 있는 상태
- 오류 상태: 마지막 수집 실패, 문서 추출 실패 등 재시도 안내

### 비고
- 지원 준비 생성 전 선행 데이터가 부족하면 어떤 데이터가 부족한지 안내해야 한다.

## SCR-04 GitHub 연결 및 URL 입력

### 목적
public repository 조회 또는 OAuth 확장 연결을 시작한다.

### 권장 라우트
- `/portfolio/github`

### 주요 구성요소
- 현재 GitHub 연결 상태
- GitHub OAuth 연결 버튼
- GitHub 프로필 또는 repository URL 입력 필드
- `public repository만 조회 가능` 안내 문구
- private repository 접근을 위한 추가 동의 설명
- scope 부족 시 재동의 CTA

### 주요 액션
- URL 기반 public 조회
- OAuth 연결 또는 재연결
- 추가 scope 동의

### 상태
- 미연결
- public 조회만 가능
- private 접근 가능
- scope 부족
- API 실패

### 검증/예외
- 잘못된 GitHub URL 형식 차단
- URL 조회는 public만 가능함을 명시
- private repository가 필요하면 OAuth 추가 동의 필요
- GitHub API 오류 시 재시도 제공

### 완료 후 이동
- repository 목록 조회 성공 시 SCR-05

## SCR-05 repository 선택 및 commit 수집

### 목적
사용할 repository를 선택하고 commit 수집을 시작한다.

### 권장 라우트
- `/portfolio/repositories`

### 주요 구성요소
- repository 목록 테이블 또는 카드
- 필터: 최근 2년 활동 우선 정렬
- `전체 보기` 토글
- visibility 배지
- 선택 체크박스
- `선택 저장`
- `commit 수집 시작`

### 표시 규칙
- 최근 2년 활동 repository를 먼저 노출하되 전체 목록 접근은 가능해야 한다.
- visibility가 private인 항목은 추가 동의가 있는 경우에만 노출한다.
- PR, Issue 관련 선택 UI는 이번 버전에서 제공하지 않는다.

### 상태
- 목록 로딩
- 목록 없음
- 일부 조회 성공/일부 실패
- commit 수집 진행 중
- commit 수집 완료
- commit 수집 실패

### 검증/예외
- 선택 없이 저장 또는 수집 시작 시 안내
- scope 부족으로 private 항목을 불러오지 못하면 재동의 CTA 제공
- GitHub rate limit 또는 API 실패 시 재시도 안내

### 완료 후 이동
- 문서 업로드가 아직 없으면 SCR-06 권장
- 포트폴리오 입력이 충분하면 SCR-07 또는 SCR-08 이동 가능

## SCR-06 문서 업로드 및 추출 상태

### 목적
이력서, 수상기록, 증빙 문서를 업로드하고 텍스트 추출 상태를 확인한다.

### 권장 라우트
- `/portfolio/documents`

### 주요 구성요소
- 업로드 드롭존
- 허용 형식 안내: PDF, DOCX, MD
- 제한 안내: 파일당 10MB, 최대 5개
- 업로드 목록
- 문서 유형 선택: 이력서, 수상기록, 증빙, 기타
- 추출 상태 배지: 대기, 성공, 실패
- 실패 사유 안내
- 재업로드/삭제/덮어쓰기 버튼

### 상태
- 빈 상태
- 업로드 진행 중
- 추출 대기
- 추출 성공
- 추출 실패
- 파일 제한 초과
- 중복 감지

### 검증/예외
- 허용하지 않는 형식 차단
- 10MB 초과 차단
- 5개 초과 차단
- 동일 문서 감지 시 덮어쓰기 확인
- 스캔 PDF는 추출 실패 가능성 사전 안내
- OCR은 기본 제공 아님을 명시

### 완료 후 이동
- SCR-07 또는 SCR-08

## SCR-09 지원 준비 생성/수정

### 목적
회사, 직무, 지원 유형과 사용할 source를 선택한다.

### 권장 라우트
- `/applications/new`
- `/applications/{applicationId}/edit`

### 주요 구성요소
- 회사명 입력
- 직무명 입력
- 지원 유형 입력
- source repository 선택
- source document 선택
- 기본 분량 옵션: short, medium, long
- 저장 버튼

### 상태
- 신규 생성
- 수정
- source 부족
- 저장 성공
- 저장 실패

### 검증/예외
- 직무명은 필수
- source를 하나도 선택하지 않으면 자소서 생성 단계 진입 전 경고
- 저장 성공 후 `draft` 또는 `ready` 상태를 보여준다.

### 완료 후 이동
- SCR-10

## SCR-10 자소서 문항 입력 및 생성 결과

### 목적
직무별 기본 템플릿을 불러오고, 사용자가 문항을 추가 또는 수정하여 자소서 초안을 생성한다.

### 권장 라우트
- `/applications/{applicationId}/self-intro`

### 주요 구성요소
- 직무별 기본 템플릿 불러오기
- 문항 입력 리스트
- 문항 직접 추가
- 문항 재정렬
- 문항별 생성 결과 카드
- 분량 옵션 표시
- `초안 생성`
- `재생성`
- `저장`

### 상태
- 초기: 문항 없음
- 템플릿 적용됨
- 생성 중
- 부분 성공
- 전체 실패
- 결과 저장 완료

### 검증/예외
- 문항은 한 번에 최대 10개
- 포트폴리오가 부족하면 생성 전 안내
- 회사별 특수 문항은 수동 추가
- 중복 생성 요청은 차단 또는 큐 처리 안내
- 10초 이상 지연 시 재시도 또는 잠시 후 재확인 안내

### 완료 후 이동
- 질문 생성: SCR-11
- 목록 복귀: SCR-08

## SCR-11 면접 질문 생성

### 목적
자소서와 포트폴리오를 기반으로 면접 질문 세트를 생성한다.

### 권장 라우트
- `/applications/{applicationId}/question-sets/new`

### 주요 구성요소
- 기준 application 정보 요약
- 질문 카테고리 선택 또는 기본값
- 질문 수 입력
- 난이도 선택
- 생성 버튼
- 생성 결과 미리보기

### 상태
- 기본
- 생성 중
- 생성 성공
- 생성 실패
- 입력 데이터 부족

### 검증/예외
- 자소서가 없으면 먼저 SCR-10으로 유도
- 질문 수는 최대 20개
- 품질 저하 가능 시 사전 경고
- 생성 실패 시 재시도 제공

### 완료 후 이동
- SCR-12

## SCR-12 질문 세트 상세

### 목적
생성된 질문을 검토하고 실제 면접 세션 시작 전 최종 조정한다.

### 권장 라우트
- `/interview/question-sets/{questionSetId}`

### 주요 구성요소
- 질문 리스트
- 질문 카테고리/난이도
- 개별 삭제
- 수동 질문 추가 입력(question text, question type, difficulty level)
- 선택 질문 수 표시
- `모의 면접 시작`

### 상태
- 정상
- 수정 중
- 저장 중
- 활성 세션 존재

### 검증/예외
- 질문 세트 편집은 세션 시작 전까지만 허용한다.
- 수동 질문 추가 1차 범위는 독립 질문만 허용하고 `follow_up`와 `parentQuestionId`는 받지 않는다.
- 질문 삭제는 hard delete로 처리하고, 삭제 후 남은 질문 순서는 서버가 자동 재정렬한다.
- 질문 추가/삭제 시 질문 순서는 사용자가 직접 조정하지 않고 서버 응답 기준으로 다시 그린다.
- 실제 세션 시작 시 질문 수가 3개 이상 20개 이하인지 검증
- 활성 세션이 이미 있으면 새 세션 시작 대신 기존 세션 이동 유도

### 완료 후 이동
- SCR-13

## SCR-13 텍스트 모의 면접 세션

### 목적
질문을 순차 제시하고 답변을 저장하며 세션 상태를 관리한다.

### 권장 라우트
- `/interview/sessions/{sessionId}`

### 주요 구성요소
- 현재 질문 카드
- 답변 입력 textarea
- 글자 수 카운터
- `제출`
- `건너뛰기`
- `일시정지`
- 진행률
- 남은 질문 수
- 자동 저장 상태
- 세션 상태 배지

### 세션 상태 표기
- ready
- in_progress
- paused
- completed
- feedback_completed

### 상태
- 진입 직후 복원 가능
- 진행 중
- 일시정지
- 자동 일시정지
- 완료 처리 중
- 완료

### 검증/예외
- 일반 답변은 50자 미만 제출 차단
- 일반 답변은 1000자 초과 차단
- `건너뛰기`는 별도 버튼으로 처리
- 세션 진입 시 `GET /interview/sessions/{sessionId}` 한 번으로 현재 질문, 진행률, 남은 질문 수, `resumeAvailable`, `lastActivityAt`를 구성한다.
- `일시정지`와 `재개`는 명시적 액션으로 처리한다.
- v1 자동 일시정지는 세션 진입, 답변 제출, 재개 요청 시점에 `lastActivityAt` 기준으로 평가한다.
- 새로고침/브라우저 이탈 후 30분 이내 복원 안내
- 30분 이상 미입력 시 자동 일시정지
- 완료된 세션에는 추가 입력 불가

### 완료 후 이동
- 세션 완료: SCR-14
- 일시정지 후 복귀: 동일 세션
- 활성 세션 재진입: SCR-15 또는 직접 세션 라우트

## SCR-14 면접 결과 리포트

### 목적
세션 종료 후 피드백 리포트를 보여준다.

### 권장 라우트
- `/interview/sessions/{sessionId}/result`

### 주요 구성요소
- 총평
- 답변 완성도
- 기술 정확도
- 구체성
- 개선 제안
- 태그 요약
- 질문별 상세 피드백
- `다시 보기`
- `히스토리로 이동`

### 상태
- 피드백 생성 중
- 피드백 생성 완료
- 피드백 생성 실패 또는 미준비 상태 재확인 가능

### 검증/예외
- 피드백 생성 실패 시 세션 기록은 유지
- `POST /interview/sessions/{sessionId}/complete`에서 `502`, `503`이 반환돼도 세션 기록은 유지한다.
- 결과가 아직 준비되지 않았으면 대기 또는 재확인 안내를 노출한다.
- v1의 재시도 CTA는 `POST /interview/sessions/{sessionId}/complete` 재전송이 아니라 `GET /interview/sessions/{sessionId}/result` 재조회 또는 히스토리 재진입이다.

## SCR-15 면접 히스토리 목록

### 목적
이전 세션 목록과 현재 활성 세션을 보여준다.

### 권장 라우트
- `/interview/history`

### 주요 구성요소
- 활성 세션 배너
- 세션 목록
- 상태 필터
- 질문 세트 기준 정렬
- 최근 피드백 점수 요약
- 상세 보기 버튼

### 상태
- 빈 상태
- 활성 세션 있음
- 목록 로딩
- 조회 실패

### 비고
- 활성 세션이 있으면 가장 위에 고정해서 보여주고 재개 버튼을 제공한다.
- 목록 데이터는 `GET /interview/sessions` 하나로 조회한다.
- 서버는 현재 사용자 세션만 반환하고, 활성 세션(`in_progress`, `paused`)이 있으면 응답 배열의 가장 앞에 둔다.
- 활성 세션 다음의 과거 세션은 `startedAt` 최신순으로 노출한다.
- SCR-15의 상태 필터와 화면 정렬은 별도 백엔드 query parameter가 잠기기 전까지 v1 목록 응답 범위 안에서 처리한다.
- `completed` 상태지만 결과가 아직 준비되지 않은 세션은 결과 재확인 흐름으로 연결한다.

## 7. 화면 간 주요 전환 규칙

### 7.1 선행 조건 전환
- 포트폴리오 데이터가 없으면 자소서 화면으로 직접 진입하지 못한다.
- 자소서가 없으면 질문 생성 화면으로 직접 진입하지 못한다.
- 질문 세트가 없으면 세션 시작 화면으로 직접 진입하지 못한다.

### 7.2 세션 우선 전환
- 활성 세션이 존재하면 새 세션 생성보다 기존 세션 복귀를 우선한다.
- `paused`도 활성 세션으로 본다.
- 세션이 `paused`이면 복귀 CTA를 우선 노출한다.

### 7.3 오류 후 전환
- 추출 실패 문서는 전체 흐름을 막지 않되, 품질 저하 가능성을 명시한다.
- GitHub scope 부족은 연결 관리 화면으로 되돌린다.
- 자소서 생성 실패와 질문 생성 실패는 현재 화면에서 재시도 가능해야 한다.

## 8. 프론트 구현 체크포인트

- 파일 업로드 제약은 프론트에서 1차 차단하고 서버 검증 결과도 그대로 노출한다.
- 세션 상태 전이는 서버 응답을 기준으로 처리한다.
- `건너뛰기`와 `일반 제출`은 동일한 answers API를 사용하되 `isSkipped`와 `answerText` 검증 규칙으로 구분한다.
- 로그인 제공자 충돌 안내는 단순 에러 토스트가 아니라 다음 행동이 보이는 UI로 만든다.
- 화면의 라벨은 내부 테이블명보다 사용자 용어를 우선 사용한다.
