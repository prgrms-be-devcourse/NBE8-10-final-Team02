# AI 기술 면접 연습 플랫폼 비기능 요구사항

이 문서는 MVP 구현과 운영에 필요한 상위 품질 기준을 잠그는 문서다.
세부 규칙은 아래 원본 문서가 우선한다.
- 보안: `docs/security.md`
- 테스트: `docs/testing.md`
- 관측성: `docs/observability.md`
- 오류: `docs/error-policy.md`
- API 계약: `openapi.yaml`
- 저장 구조: `docs/db/schema.md`

## 1. 문서 목적과 적용 범위

- 기능 요구사항을 안정적으로 구현하기 위한 품질 기준을 정의한다.
- MVP 기준 백엔드, API, DB, 외부 연동, 파일 처리, AI 생성, 운영 환경 전반에 적용한다.
- 본 문서는 방향과 수치 기준을 잠그고, 세부 실행 절차는 하위 운영 문서에 위임한다.

## 2. 상위 원칙

- 기능 범위와 충돌하면 `docs/requirements.md`를 우선한다.
- 인증, 인가, 데이터 소유권 검증은 모든 사용자 전용 기능의 기본 전제다.
- 외부 연동 실패는 전체 시스템 장애가 아니라 부분 실패로 식별 가능해야 한다.
- 업로드, 동기화, 생성은 요청 성공과 후속 처리 성공을 분리해 상태로 관리한다.
- 품질 기준은 화면, API, DB, 운영 문서에 동일하게 드러나야 한다.

## 3. 서비스 수준 목표

### 3.1 가용성
- MVP 운영 환경의 월간 가용성 목표는 99.0% 이상이다.
- 계획된 점검은 사전 공지 또는 내부 운영 시간에 제한한다.

### 3.2 복구 목표
- 애플리케이션 프로세스 재시작 후 핵심 API 재가동 목표는 15분 이내다.
- 단순 배포 실패 시 이전 안정 버전 롤백 시작 목표는 30분 이내다.
- DB 복구 절차는 `docs/deployment-runbook.md`와 실제 운영 문서에서 구체화한다.

### 3.3 성능 측정 기준
- 응답 시간 평가는 p95를 기본 기준으로 사용한다.
- 외부 연동 없는 일반 조회 API는 p95 1초 이내를 목표로 한다.
- 일반 저장/수정/삭제와 면접 답변 저장은 p95 2초 이내를 목표로 한다.
- GitHub repository 목록 조회는 p95 3초 이내를 목표로 한다.
- 문서 업로드 초기 응답은 p95 5초 이내를 목표로 한다.
- 자소서 생성은 p95 30초, 면접 질문 생성은 p95 20초, 면접 결과 생성은 p95 30초 이내를 목표로 한다.

## 4. 보안 및 인증 기준

### 4.1 인증
- 보호된 API는 인증된 사용자만 접근할 수 있어야 한다.
- OAuth2 로그인 제공자는 GitHub, Google, Kakao로 제한한다.
- 토큰, secret, 외부 credential은 로그에 남기지 않는다.

### 4.2 인가
- 사용자는 자신의 문서, GitHub 연결, application, 면접 세션, 결과 데이터에만 접근할 수 있어야 한다.
- resource ID를 직접 입력해도 타인 데이터 조회/수정이 되면 안 된다.
- 수정, 삭제, 결과 조회는 소유권 검증을 필수로 수행한다.

### 4.3 private repository 접근
- GitHub URL 입력만으로는 public repository만 조회한다.
- private repository 접근은 GitHub OAuth 연동과 추가 동의가 있는 경우에만 허용한다.
- scope 부족, 접근 불가, rate limit, token 만료는 구분 가능한 오류로 처리한다.

## 5. 데이터 보호와 무결성

### 5.1 수집 최소화
- 서비스 제공에 필요한 최소 범위의 사용자 정보만 저장한다.
- 생성형 AI 입력에는 불필요한 개인정보를 포함하지 않는다.

### 5.2 분리와 소유권
- 사용자별 포트폴리오, 생성 결과, 면접 결과는 논리적으로 분리되어야 한다.
- 다른 사용자 데이터가 AI 컨텍스트에 섞이면 안 된다.

### 5.3 저장 무결성
- unique 제약과 참조 무결성은 실제 스키마에 반영되어야 한다.
- 동일 provider 계정, repository, commit, answer-tag 중복 저장은 허용하지 않는다.
- 세션 종료 전 결과 완료 상태가 되면 안 된다.

### 5.4 삭제와 보관
- 계정 탈퇴 시 데이터 삭제 정책은 요구사항과 스키마 문서를 따른다.
- 로그 보관 기간과 개인정보 보관 기간은 운영 정책으로 별도 관리하되, 서비스 문서에 반영되어야 한다.

## 6. 파일 업로드와 문서 처리 기준

- 지원 형식은 PDF, DOCX, MD로 고정한다.
- 파일 크기는 파일당 최대 10MB, 사용자당 기본 최대 5개다.
- 프론트 사전 검증과 서버 최종 검증은 같은 기준을 사용한다.
- 문서 업로드 성공과 텍스트 추출 성공은 별도 상태로 관리한다.
- 암호화 문서, 추출 불가 문서, 비정상적으로 짧은 추출 결과는 실패 또는 품질 저하 상태로 식별 가능해야 한다.

## 7. 외부 연동과 AI 생성 신뢰성

### 7.1 OAuth2 / GitHub 연동
- OAuth2 콜백 실패, provider 장애, GitHub scope 부족, rate limit 초과를 구분된 오류로 남겨야 한다.
- repository 조회 실패와 commit 동기화 실패는 별도 상태로 관리 가능해야 한다.

### 7.2 AI 생성 실패 처리
- timeout, 빈 응답, malformed JSON, schema violation, 질문 수/점수 불일치 같은 비정상 결과는 정상 저장하면 안 된다.
- 실패 사유를 남겨 재시도 가능 여부를 판단할 수 있어야 한다.
- 어떤 모델과 어떤 전략이 사용됐는지 추적 가능해야 한다.

### 7.3 재시도 정책
- 외부 API 호출은 connect/read timeout을 명시적으로 설정한다.
- 무한 재시도는 금지한다.
- MVP 기준 AI 자동 재시도는 아래로 고정한다.
  - 자소서 생성: 최대 2회
  - 면접 질문 생성: 최대 2회
  - 꼬리 질문 생성: 최대 1회
  - 면접 평가 생성: 최대 2회
  - 면접 요약 생성: 최대 1회
- fallback model 자동 전환은 MVP 기본값으로 비활성화한다.

## 8. 세션과 상태 관리 기준

- 사용자당 동시에 진행 가능한 활성 면접 세션은 1개다.
- 일반 답변 길이는 50자 이상 1000자 이하로 제한한다.
- 30분 이상 답변이 없으면 세션을 자동 일시정지하고 재개 가능해야 한다.
- 생성 작업과 세션 상태는 `pending`, `in_progress`, `paused`, `completed`, `failed`처럼 화면과 API에서 일관되게 보이도록 관리한다.

## 9. 테스트와 품질 게이트

- 핵심 도메인 서비스는 단위 테스트가 있어야 한다.
- API 입력 validation, 인가, 오류 응답은 통합 또는 API 테스트로 검증 가능해야 한다.
- DB 제약, 매핑, 트랜잭션 경계는 통합 테스트로 확인해야 한다.
- 외부 연동 어댑터는 mock 또는 sandbox 기반 테스트가 가능해야 한다.
- main 또는 develop 병합 전 자동 테스트가 통과해야 한다.
- 실패한 빌드는 병합 기준을 통과할 수 없어야 한다.

필수 검증 흐름은 아래를 포함한다.
- OAuth2 로그인 성공/실패
- GitHub repository 조회와 commit 동기화 성공/실패
- 문서 업로드와 추출 성공/실패
- 자소서 생성, 질문 생성, 면접 결과 생성 성공/실패
- 면접 세션 생성, 답변 저장, 자동 일시정지, 종료, 결과 조회

## 10. 로그, 모니터링, 추적성 기준

- 모든 주요 요청 흐름은 requestId 또는 traceId로 추적 가능해야 한다.
- 구조화 로그를 기본으로 하며 로그 레벨을 구분한다.
- 이메일, 토큰, 원문 문서 전체, 사용자 답변 전체는 기본 운영 로그에 그대로 남기지 않는다.
- 아래 지표는 최소 모니터링 대상이다.
  - 로그인 성공/실패 수
  - GitHub 연동 성공/실패 수
  - 문서 업로드 및 추출 성공/실패 수
  - 자소서 생성, 질문 생성, 평가 생성 성공/실패 수
  - 세션 시작/종료 수
  - 외부 API timeout 및 error 비율
- 애플리케이션 헬스체크와 DB 연결 상태는 확인 가능해야 한다.
- 오류율, GitHub 연동 실패율, 문서 추출 실패율, AI 생성 실패율 급증은 경고 대상이다.

## 11. 배포와 운영 기준

- local, dev, prod 환경은 분리된 설정을 사용한다.
- 애플리케이션은 재현 가능한 빌드 결과물로 배포되어야 한다.
- Docker 기반 실행 환경을 기본 표준으로 둔다.
- 배포 시 설정 누락, migration 실패, health check 실패를 감지할 수 있어야 한다.
- 스키마 변경은 migration 또는 명시적 DDL 관리 절차 없이 수동 반영하지 않는다.
- 운영 DB 직접 수정은 긴급 복구를 제외하고 금지한다.

## 12. 유지보수성과 확장성

- 인증, GitHub 연동, 문서 처리, 자소서 생성, 면접 세션 도메인은 책임이 분리된 구조여야 한다.
- 외부 연동과 도메인 로직은 직접 강결합하지 않도록 추상화 계층을 둔다.
- PR/Issue 수집, 음성 면접, 결제, 통계 기능이 추가되어도 핵심 엔티티와 상태 체계를 전면 재작성하지 않도록 설계한다.
