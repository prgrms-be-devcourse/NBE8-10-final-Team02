# 요구사항 변경 반영 후 검토 메모

이 문서는 최신 요구사항 변경안을 문서 세트에 반영하면서 발견한 충돌 지점과, 이번 버전에서 추천 기준으로 정리한 결론을 기록한다.
구현자는 이 문서와 `docs/requirements.md`, `openapi.yaml`을 함께 본다.

## 1. GitHub private repository 지원 범위

### 정리 결과
- GitHub URL 입력만으로는 public repository만 조회한다.
- private repository는 GitHub OAuth 추가 동의와 적절한 scope가 있을 때만 접근을 허용한다.
- MVP 기본 수집 대상은 repository와 사용자 본인 commit이며, private repository를 허용하더라도 PR, Issue 수집까지 동시에 열지 않는다.

### 이유
- private repository 허용 자체가 보안, 권한 오류, 테스트 범위를 크게 늘리므로 기능 범위를 commit 중심으로 제한해야 MVP 난이도를 통제할 수 있다.
- URL 입력과 OAuth 연동을 같은 정책으로 취급하면 권한 모델이 불명확해진다.

## 2. PR, Issue 수집 문구의 모호성

### 정리 결과
- 요구사항 초안의 `필요 시 PR, 이슈 등 활동 내역 수집` 문구는 canonical 기준에서 후속 확장으로 정리했다.
- 현재 버전의 저장 구조와 화면 흐름은 commit 중심 입력에 맞춘다.
- **[후속 확장 구현 진행 중 — feat/github-metadata-pipeline]**
  - GitHub Metadata 파이프라인 (Issue/PR/Review 수집 → AI 프롬프트 병합) 구현 시작.
  - OAuth scope 확인 결과: `public_repo` 으로 PR Review 포함 모든 메타데이터 접근 가능 — 추가 동의 불필요.
  - 설계 문서: `docs/personal-design/github-metadata-pipeline.md`

### 이유
- PR, Issue까지 포함하면 저장 구조, 권한 scope, 화면 선택 UI, 동기화 정책이 모두 다시 커진다.
- 현재 사용자 흐름과 와이어프레임은 commit 중심 흐름일 때 가장 일관적이다.

## 3. 최근 2년 repository 우선 수집 규칙

### 정리 결과
- `최근 2년 이내 활동 repo 우선 수집`은 하드 필터가 아니라 우선 정렬 기준으로 적용한다.
- 최근 활동 repository를 먼저 보여주되, 필요 시 전체 목록을 조회할 수 있게 한다.

### 이유
- 하드 필터로 적용하면 오래된 대표 프로젝트가 누락될 수 있다.
- 우선 정렬로 해석하면 UX와 구현 난이도를 모두 안정적으로 가져갈 수 있다.

## 4. 동일 문서 재업로드 정책

### 정리 결과
- 같은 문서를 다시 업로드하면 사용자에게 덮어쓰기 여부를 확인한다.
- 구현 권장안은 파일 해시 기반 중복 감지 후 `덮어쓰기` 또는 `취소`를 선택하게 하는 방식이다.

### 이유
- 파일명만으로 중복을 판단하면 오검출 가능성이 크다.
- 자동 덮어쓰기는 사용자의 저장 이력 기대와 충돌할 수 있다.

## 5. 같은 이메일의 다른 소셜 로그인 처리

### 정리 결과
- 자동 병합은 하지 않는다.
- 기존 계정 안내 후 동일 제공자로 로그인하거나, 로그인 이후 별도 계정 연결 절차로 유도한다.

### 이유
- 제공자별 이메일 검증 여부가 다를 수 있어 자동 병합은 계정 탈취 위험을 만든다.
- 명시적 계정 연결 흐름이 보안과 사용자 이해 측면에서 더 안전하다.

## 6. 최소 답변 길이 50자 규칙

### 정리 결과
- 최소 50자 규칙은 `건너뛰기 아닌 일반 답변`에만 적용한다.
- `isSkipped=true`인 경우에는 짧은 메모나 빈 문자열도 허용한다.

### 이유
- 모든 답변에 50자를 강제하면 실제 면접 흐름과 맞지 않는 상황이 생긴다.
- 건너뛰기와 부실 답변을 같은 규칙으로 처리하면 UX가 나빠진다.

## 7. 스캔 PDF OCR 처리

### 정리 결과
- 스캔 PDF는 추출 실패 가능 항목으로 분리한다.
- OCR은 MVP 기본 범위에 넣지 않고 추후 확장으로 둔다.

### 이유
- OCR은 정확도, 비용, 처리 시간, 운영 복잡도를 모두 추가한다.
- 지금 단계에서는 업로드 실패와 추출 실패를 분리해서 안내하는 편이 더 안전하다.

## 이번 반영에서 확정한 값 요약

- 업로드 허용 형식: PDF, DOCX, MD
- 업로드 용량 제한: 파일 1개당 10MB
- 업로드 파일 수: 최대 5개
- private repository: OAuth 추가 동의 시 허용
- 자소서 문항 수: 최대 10문항
- 면접 질문 생성 수: 최대 20개
- 면접 세션 질문 수: 3개 이상 20개 이하
- 답변 길이: 50자 이상 1000자 이하, 단 건너뛰기 예외
- 회원 탈퇴: 30일 유예 후 완전 삭제
