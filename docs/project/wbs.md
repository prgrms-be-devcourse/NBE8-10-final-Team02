# AI 기술 면접 연습 플랫폼 WBS

## 1. 문서 목적

이 문서는 최신 `docs/requirements.md`, `docs/project/open-items.md`, `openapi.yaml`,
`docs/frontend/screen-spec.md`를 기준으로 MVP 작업 단위, 선행 관계, 산출물,
권장 담당, 완료 기준을 정리한 실행용 WBS다.

이전 버전이 큰 작업 묶음 중심이었다면, 이번 버전은 실제 착수 가능한 단위로 더 잘게 나눈다.

## 2. 프로젝트 전제

- 팀 구성: 4인 백엔드 중심
- 목표 범위: 1차 MVP
- 기준 기간: 4주
- 병행 원칙:
  - 문서 기준이 없는 작업은 구현 전에 문서를 먼저 잠근다.
  - API, DB, 화면 상태가 같이 바뀌는 변경은 같은 PR에서 묶는다.
  - private repository 허용은 OAuth scope 처리까지 포함되므로 GitHub 연동 작업에서 별도 하위 항목으로 본다.

## 3. MVP 포함/제외 범위

### 3.1 포함
- OAuth2 소셜 로그인
- GitHub URL 기반 public repository 조회
- GitHub OAuth 기반 private repository 조회
- repository 선택 및 사용자 본인 commit 수집
- 문서 업로드, 텍스트 추출, 중복 문서 덮어쓰기 확인
- 지원 준비 생성
- 직무별 템플릿 기반 자소서 생성
- 자소서 기반 면접 질문 생성
- 텍스트 모의 면접
- 결과 리포트와 히스토리 조회

### 3.2 제외
- PR, Issue 수집
- OCR 기반 스캔 PDF 처리
- 음성/영상 면접
- 결제, 광고, 토큰 시스템
- 커뮤니티, 게임화, 대규모 통계

## 8. 병목 위험과 대응

### 8.1 GitHub 권한 범위
- 위험: private repository scope 처리에서 오류가 나면 포트폴리오 핵심 흐름이 막힌다.
- 대응: URL 기반 public 조회를 먼저 완료하고, OAuth 확장을 분리 구현한다.

### 8.2 문서 추출 품질
- 위험: 스캔 PDF를 처리하지 못하면 사용자가 업로드 실패로 오해할 수 있다.
- 대응: 업로드 성공과 추출 실패를 분리해 저장하고 화면에서 명확히 안내한다.

### 8.3 AI 응답 지연
- 위험: 자소서/질문/피드백 생성이 느리면 세션이 멈춘 것처럼 보인다.
- 대응: 타임아웃, 재시도 안내, 비동기 상태 노출을 먼저 설계한다.

### 8.4 세션 상태 복잡도
- 위험: 활성 세션 1개 제한, skip, pause, complete를 같이 다루면 상태 버그가 나기 쉽다.
- 대응: 상태 전이 테스트를 별도 작업으로 잡고 완료 전 수동 시연까지 포함한다.

## 9. 이번 버전에서 완료로 볼 기준

아래가 모두 맞으면 이번 WBS를 구현 기준으로 사용할 수 있다.

- 요구사항 변경안의 숫자 제약이 작업 단위에 반영되어 있다.
- GitHub public/private 흐름이 분리되어 있다.
- 화면 상태 문서와 작업 항목의 선행 관계가 맞는다.
- 세션 상태와 테스트 항목이 따로 드러나 있다.
- 작업을 누가 먼저 시작해야 하는지가 표만 보고 보인다.
